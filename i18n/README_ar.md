## 🌟 حول المشروع

هذا المشروع عبارة عن نظام خبير لدعم اتخاذ القرار لتداول صناديق الاستثمار المتداولة (ETF)، باستخدام نهج الذكاء الاصطناعي الهجين ثلاثي الأبعاد. تم تصميمه لتقديم تحليل شامل وقوي من خلال الجمع بين وجهات نظر متعددة للذكاء الاصطناعي.

### 🚀 استراتيجية الرمز المزدوج (التحليل مقابل التداول)
يستخدم النظام نهجاً مبتكراً لزيادة دقة النماذج إلى أقصى حد:
- **تحليل عالي الدقة**: تقوم نماذج الذكاء الاصطناعي بتحليل **المؤشرات المرجعية العالمية** (`^NDX` لناسداك، `CL=F` للنفط الخام WTI). توفر هذه المؤشرات تاريخاً أطول واتجاهات "أنقى"، بدون الضوضاء المتعلقة بساعات التداول أو رسوم صناديق الاستثمار المتداولة.
- **تنفيذ صناديق الاستثمار المتداولة**: يتم وضع الأوامر الحقيقية على الرموز المقابلة في **Trading 212** (`SXRV.DE`، `CRUDP.PA`)، باستخدام **الأسعار المباشرة من T212** (عبر واجهة برمجة تطبيقات المراكز) لتحديد حجم المركز. تتم مزامنة حالة المحفظة مباشرة من T212 (`sync_state_from_t212()`)، ويتم حقن الأسعار المباشرة في مسار التحليل (`_inject_t212_live_price()` في `src/data.py`).

### 🧠 محرك الذكاء الاصطناعي الهجين
يدمج النظام أحد عشر إشارة مميزة:
1.  **النموذج الكمي الكلاسيكي**: مجموعة من RandomForest/GradientBoosting/LogisticRegression تم تدريبها على المؤشرات الفنية والاقتصادية الكلية.
2.  **TimesFM 2.5 (أبحاث جوجل)**: نموذج أساسي متطور للتنبؤ بالسلاسل الزمنية.
3.  **TensorTrade / PPO (التعلم المعزز)**: وكيل التعلم المعزز (stable-baselines3) يقوم بتدريب سياسة PPO في بيئة تداول Gymnasium مخصصة مع الاستمرارية عبر الدورات.
4.  **نموذج Oil-Bench (Gemma 4 12B (Unsloth))**: نموذج متخصص في الطاقة يدمج البيانات الأساسية من **EIA** (المخزونات، الواردات، استخدام المصافي) والمشاعر لتداول WTI.
5.  **النموذج اللغوي الكبير النصي (Gemma 4 12B (Unsloth))**: تحليل سياقي للبيانات الأولية، أخبار في الوقت الفعلي عبر مهارة **AlphaEar**، ودمج **بحث الويب الماكرو اقتصادي** الديناميكي.
6.  **النموذج اللغوي الكبير المرئي (Gemma 4 12B (Unsloth))**: تحليل مباشر للمخططات الفنية (`enhanced_trading_chart.png`).
7.  **تحليل المشاعر**: تحليل هجين يجمع بين Alpha Vantage والاتجاهات "الساخنة" من **AlphaEar** (Weibo, WallstreetCN).
8.  **البيانات اللامركزية (Hyperliquid)**: تحليل المشاعر المضاربة على النفط (WTI) عبر *Funding Rate* و *Open Interest*.
9.  **نموذج Vincent Ganne**: تحليل جيوسياسي ومتعدد الأصول (WTI، برنت، الغاز، DXY، المتوسط المتحرك 200) لاكتشاف القيعان الماكرو اقتصادية.
10. **نموذج Grebenkov**: نموذج رياضي لتتبع الاتجاه تمت معايرته لتحليل الأصول المتعددة باستخدام تعادل المخاطر اللاأدري (Agnostic Risk Parity).
11. **محرك الدمج الهجين**: النموذج الوصفي (Meta-model) الذي ينسق الأوزان الديناميكية والإجماع المعرفي عبر جميع النماذج الفرعية.

الهدف هو الوصول إلى قرار نهائي (`BUY`، `SELL`، `HOLD`) مع أولوية مطلقة لـ **الدقة أولاً**.

### 🧘 فلسفة القرار: "الحذر المعرفي"
على عكس خوارزميات التداول الكلاسيكية التي تصاب بالذعر بمجرد انفجار التقلبات، يطبق هذا النظام نهج المستثمر المستنير:
- **إجماع قوي مطلوب**: قد يطلق النموذج الكمي (الكلاسيكي) إنذاراً (`SELL`)، ولكن إذا ظلت النماذج المعرفية (النص، الرؤية، TimesFM) محايدة، سيفضل النظام `HOLD`.
- **مُرشّح الثقة**: يتم التحقق من قرار الحركة (شراء أو بيع) فقط إذا تجاوزت الثقة الإجمالية حد الأمان (عادة 40٪). دون ذلك، يعتبر النظام الإشارة كـ "ضوضاء" ويبقى في وضع الاستعداد.
- **حماية رأس المال**: في وضع الخطر `VERY_HIGH`، يعمل `HOLD` كدرع. يمنع الدخول في سوق غير مستقر ويتجنب الخروج المبكر في تصحيح فني بسيط إذا كانت الأساسيات (الأخبار/الرؤية/Hyperliquid) لا تؤكد انهياراً وشيكاً.

### ✨ الميزات الرئيسية

- **نهج الرمز المزدوج**: تحليل المؤشر، وتداول صندوق الاستثمار المتداول (ETF).
- **الأسعار المباشرة T212**: استرداد فوري لأسعار اليورو عبر واجهة برمجة تطبيقات Trading 212 (0.2 ثانية)، مع الاحتياط إلى yfinance وذاكرة التخزين المؤقت parquet.
- **فارق برنت المؤرخ (Dated Brent Spread)**: مراقبة توتر السوق الفعلي عبر الفارق بين السعر الفوري لبرنت (المؤرخ) وعقود برنت الآجلة.
- **مرونة الشبكة**: قاطع دائرة yfinance مع متتبعات منفصلة (المعلومات مقابل التنزيل)، مهلة 10 ثوانٍ لجميع مكالمات الشبكة.
- **إبطال التخزين المؤقت التلقائي**: تكتشف ذاكرة التخزين المؤقت Parquet تلقائياً التقادم (أكثر من يومين) وتجبر التحديث. استخدم `refresh_cache.py` لمسح ذاكرة التخزين المؤقت يدوياً.
- **توازي مكالمات النماذج اللغوية الكبيرة (LLM)**: تعمل استدعاءات النماذج المستقلة (`text_llm`, `visual_llm`, `search_query`, `timesfm`, `tensortrade`, `grebenkov`) في `ThreadPoolExecutor` لتداخل استنتاج Ollama مع الإدخال/الإخراج. المسار الحرج عادة من 4-6 دقائق على وحدة المعالجة المركزية (CPU) مقابل أكثر من 10 دقائق متسلسلة.
- **ذاكرة تخزين مؤقت لاستعلام البحث لمدة 24 ساعة**: يتم تخزين استعلام البحث على الويب الذي تم إنشاؤه بواسطة LLM مؤقتاً تحت `data_cache/search_queries/<ticker>_<date>_<price-sig>.json`. يتم تحديده بالتاريخ بالإضافة إلى توقيع حركة السعر (تصنيف log2 لسعر الإغلاق + فئة RSI)، لذا فإن تغيير النظام يبطله. لا يتم **أبداً** التخزين المؤقت لاستعلامات الطوارئ (لا يمكن لفشل Ollama العابر أن يلوث ذاكرة التخزين المؤقت لمدة 24 ساعة).
- **مهلة الدورة الصارمة**: يتم تغليف كل دورة رمز بمهلة 15 دقيقة (`CYCLE_TIMEOUT_SECONDS` في `main.py`). عند انتهاء المهلة، يتم إنهاء عملية العامل بـ `shutdown(wait=False)` لتبدأ الرمز التالي على الفور؛ ويتم تطبيق HOLD على الرمز الذي انتهت مهلته. العقود المستقبلية الفردية لها مهل زمنية خاصة بها (البحث 240ثانية، المرئي 300ثانية، النصي 240ثانية، نماذج وحدة المعالجة المركزية 180ثانية لكل منها، الأخبار 90ثانية، زحف الويب 30ثانية).
- **أمان الخيوط اليتيمة (Orphan-Thread Safety)**: عند انتهاء مهلة الدورة، يتم تعيين حدث `threading.Event` لكل رمز حتى ينسحب العامل اليتيم قبل أي استدعاء `execute_t212_trade` — مما يمنع صفقات الأموال الحقيقية بعد أن يتم عرض لوحة "تم تطبيق HOLD" للمستخدم. يقوم قفل `threading.Lock` لكل رمز أيضاً بتسلسل وضع أوامر T212، مما يلغي خطر التداول المزدوج في ظل تداخل المجدول أو الاستدعاءات المزدوجة لـ `--ticker`.
- **مراقب فشل LLM**: عندما يستنفد `_query_ollama` جميع عمليات إعادة المحاولة، يحمل القاموس الاحتياطي علامة `"failed": True` حتى يتمكن منطق الإجماع النهائي من التمييز بين "اختار النموذج HOLD" و"تعطل النموذج" (يتم نشره حالياً ولكن لم يتم تصفيته — متابعة معروفة).
- **إدراك متقدم**: استخدام **Gemma 4 12B** مع **دفاع JSON ثنائي الطبقة**:
  1. **فرض المخطط من جانب الخادم** (`format: SCHEMA_*` مع `additionalProperties: false`) — الطبقة الحاملة للعبء؛ يتم تمريرها عبر معلمة `format` الخاصة بـ Ollama في كل موقع استدعاء. المخططات محددة في `src/llm_client.py` (`SCHEMA_TRADING_DECISION`، `SCHEMA_SEARCH_QUERY`، `SCHEMA_OIL_ALLOCATION`).
  2. **لاحقة مطالبة النظام الدفاعية** (`"...never add a 'thought' key."`) — سطر ثانٍ زائد عن الحاجة ولكنه غير ضار، يتم الاحتفاظ به كحزام وحمالات ضد أي تراجع مستقبلي لطبقة المخطط.

  إن الرمز المميز (token) للتفكير `<|think|>` **نشط** في جميع مطالبات نظام الإنتاج الأربعة (تمت إعادة تمكينه في 2026-06-06 على `main` بعد التحقق من فرع `think-mode`). طبقة المخطط هي ما يقوم فعلياً بتحييد عيب بقايا JSON التاريخي `<|channel>thought` (السبب الجذري لشهر مايو 2026): يؤكد `tests/check_llm_json.py` أن حالات المخطط الصارمة (`v3_schema`، `v6_schema`، `v7_schema_strict`) تنتج JSON نظيفاً حتى مع تمكين `<|think|>`، بينما تفشل المتغيرات غير الصارمة `format:json`. راجع `docs/ADR-001-think-mode-dual-layer-defence.md` للحصول على التحليل الكامل وإجراءات التراجع.
- **مشاعر الأخبار والكتل المتسلسلة (Blockchain)**: دمج **AlphaEar** و **Hyperliquid** لالتقاط المشاعر الاجتماعية والمضاربة.
- **المجدول الآلي**: نص برمجي `schedule.py` للتنفيذ المستمر (8:30 صباحاً - 6:00 مساءً) على خادم.
- **إدارة المخاطر المركزية**: يقوم `AdvancedRiskManager` بمركزية منطق مكافحة الخسارة (Stop-Loss) وإيقاف الخسارة المتحرك (Trailing Stop). لم تعد النماذج الفردية تدير هذه المخاطر، مما يضمن استراتيجية موحدة وصارمة لحماية رأس المال عبر أنظمة السوق المختلفة.
- **عقود بيانات صارمة**: تم توحيد جميع نماذج الذكاء الاصطناعي بالكامل لإرجاع فئة بيانات (dataclass) مكتوبة بقوة `ModelResult` (`signal`, `confidence`, `reasoning`)، مما يضمن التوحيد بنسبة 100٪ عبر محرك الإجماع.
- **مراجعة صحة التعليمات البرمجية**: يحافظ المشروع على معيار صحة التعليمات البرمجية **من الدرجة B** عبر عمليات التدقيق الآلية (0 كود ميت، مؤشر قابلية الصيانة مرتفع).
- **الاختبار العكسي للإنتاج**: محرك اختبار عكسي مستقل للإنتاج (`backtest_prod.py`) يعيد تشغيل إشارات الإنتاج الحقيقية ضد الأسعار الحقيقية مع رسوم T212 — لا توجد تبعيات خارجية.
- **التحكم في تفريغ التصحيح**: قم بتعيين `TRADING_DEBUG_DUMP=0` لتعطيل تفريغ فشل LLM (بحد أقصى 5 ميغابايت) `data_cache/llm_debug_fail.txt`.

### 💻 التقنيات المستخدمة

- **اللغة**: `Python 3.12+`
- **الحسابات والبيانات**: `pandas`، `numpy`، `yfinance`، `pyarrow`، `pandas_datareader`، `hyperliquid-python-sdk`
- **التعلم الآلي**: `scikit-learn`، `shap`
- **الذكاء الاصطناعي و LLM**: `requests`، `ollama`
- **استخراج بيانات الويب والبحث**: `beautifulsoup4`، `duckduckgo_search`، `crawl4ai`
- **التصور**: `matplotlib` (خلفية Agg لسلامة الخيوط)، `seaborn`، `mplfinance`
- **الأدوات المساعدة**: `tqdm`، `rich`، `python-dotenv`، `schedule`

### ⚙️ الأداء والأجهزة
تم تصميم النظام ليكون **عالي الأداء على أجهزة المستهلك** دون الحاجة إلى وحدة معالجة رسومات (GPU) مخصصة.
- **وحدة المعالجة المركزية (CPU) فقط**: يعمل استنتاج LLM (Gemma 4 12B Q4_K_M عبر Ollama) و TimesFM بالكامل على وحدة المعالجة المركزية. الإنتاجية حوالي 3-4 رموز/ثانية على وحدة معالجة مركزية حديثة ذات 8 أنوية.
- **ذاكرة الوصول العشوائي (RAM) الموصى بها**: 16 جيجابايت كحد أدنى (يُقترح 32 جيجابايت لتشغيل Gemma 4 12B بشكل مريح إلى جانب TimesFM و TensorTrade).
- **تزامن Ollama**: قم بتعيين `OLLAMA_NUM_PARALLEL=8` (موجود بالفعل في `.env` الموصى به) حتى تتمكن استدعاءات LLM المتعددة من مشاركة حمل النموذج. باستخدام ميزانية السياق الافتراضية البالغة 4 جيجابايت، تحصل الخانات المتوازية على حوالي 512 رمزاً لكل منها — سيقوم Ollama بتسلسلها إذا تجاوزت المطالبات سياق الخانة الواحدة، ولكن يُبقي `ThreadPoolExecutor` التداخل في وقت الساعة الفعلي مفيداً للخطوات المرتبطة بالإدخال/الإخراج (جلب الأخبار، زحف الويب، نماذج CPU).
- **وقت التنفيذ**: حوالي 6 إلى 9 دقائق لكل رمز على وحدة المعالجة المركزية (بدء بارد)، ومن حوالي 3 إلى 5 دقائق لكل رمز مع إصابة ذاكرة التخزين المؤقت لاستعلام البحث. يعمل الإعداد الافتراضي على رمزين (CRUDP.PA + SXRV.DE)، لذا خطط لحوالي 15 دقيقة إجمالاً.
- **مهلة الدورة**: تقتصر كل دورة رمز على 15 دقيقة (`CYCLE_TIMEOUT_SECONDS`). إذا تم التجاوز، يتم تطبيق HOLD ويبدأ الرمز التالي على الفور.
- **سرعة API**: تكامل فائق السرعة مع Trading 212 (أقل من ثانية واحدة لاسترداد السعر المباشر).

## 🚀 البدء السريع

اتبع هذه الخطوات لإعداد بيئة التطوير المحلية الخاصة بك.

### ✅ المتطلبات الأساسية

- Python 3.12+ (عبر `uv`)
- تثبيت [Ollama](https://ollama.com/) وتشغيله محلياً.
- نموذج LLM المحمّل: `ollama pull hf.co/unsloth/gemma-4-12b-it-GGUF:Q4_K_M`

### ⚙️ التثبيت

1.  **استنساخ المستودع:**
    ```sh
    git clone https://github.com/laurentvv/Trading-AI.git
    cd Trading-AI
    ```
2.  **تثبيت `uv` (إذا لم يتم ذلك من قبل):**
    انظر [astral.sh/uv](https://astral.sh/uv) لتعليمات التثبيت.

3.  **إنشاء وتنشيط البيئة الافتراضية (خطوة بالغة الأهمية):**
    يجب عليك إنشاء وتنشيط `.venv` قبل تثبيت النماذج الأساسية.
    ```bash
    uv venv
    source .venv/bin/activate  # في نظام Windows، استخدم `.\.venv\Scripts\activate.ps1`
    ```

4.  **تثبيت النماذج الأساسية:**
    قم بتشغيل البرامج النصية للتثبيت لاستنساخ النماذج إلى `vendor/` وتطبيق التصحيحات:
    ```bash
    python setup_timesfm.py
    ```

5.  **تهيئة ومزامنة البيئة:**
    ```bash
    uv sync
    ```

6.  **تثبيت المتصفحات لأبحاث الويب (Crawl4AI):**
    ```bash
    uv run python -m playwright install chromium
    ```

7.  **تكوين مفاتيح واجهة برمجة التطبيقات (API) الخاصة بك:**
    أنشئ ملف `.env` في جذر المشروع:
    ```env
    ALPHA_VANTAGE_API_KEY="YOUR_KEY"
    EIA_API_KEY="YOUR_KEY"
    ```

## 🛠️ الاستخدام

يقوم النظام بتدريب نماذجه على أحدث البيانات في كل تنفيذ قبل إعطاء قرار.

### وضع المحاكاة (التداول الورقي)

لاختبار النظام دون مخاطر برأس مال وهمي قدره 1000 يورو، استخدم علامة `--simul`. سيدير النظام تاريخاً دقيقاً لعمليات الشراء والبيع.

```sh
# تشغيل تحليل محاكى (الافتراضي: SXRV.DE - Nasdaq 100 EUR)
uv run main.py --simul

# تشغيل على النفط (WTI)
uv run main.py --ticker CRUDP.PA --simul
```

### التنفيذ الحقيقي (Trading 212)

تم الآن **دمج النظام بالكامل** مع Trading 212:
- **التحقق من المحفظة**: قبل أي إجراء، يستشير الروبوت أموالك النقدية ومراكزك الحقيقية.
- **إدارة واجهة برمجة التطبيقات (API)**: يتضمن آليات إعادة المحاولة التلقائية ضد حدود الطلب (Rate Limiting).

```sh
# تشغيل التحليل مع التنفيذ الحقيقي (تجريبي أو حقيقي وفقاً لـ .env)
uv run main.py --t212
```

## 🧪 الاختبار العكسي للإنتاج

يتضمن النظام **محرك اختبار عكسي مستقل للإنتاج** (`backtest_prod.py`) يقوم بإعادة تشغيل إشارات الإنتاج الفعلية من `logs_prod/trading_journal.csv` مقابل الأسعار الحقيقية من ملفات Parquet في `data_cache/`.

### الميزات
- **إشارات حقيقية**: يعيد تشغيل القرارات الدقيقة لمحرك الدمج الهجين المكون من 11 نموذجاً.
- **أسعار حقيقية**: يستخدم بيانات OHLCV الحقيقية لصناديق الاستثمار المتداولة (SXRV.DE، CRUDP.PA) — لا توجد وكلاء (proxies) أمريكية.
- **رسوم T212**: يحاكي نموذج الرسوم بنسبة 0.1٪ لكل عملية تداول في Trading 212.
- **مقارنة الأساس**: يحسب أداء الشراء والاحتفاظ تلقائياً كمعيار.
- **المقاييس**: نسبة شارب (Sharpe Ratio)، أقصى تراجع (Maximum Drawdown)، معدل الفوز (Win Rate)، ألفا (Alpha)، العائد الإجمالي لكل رمز.

### الاستخدام

```bash
uv run python backtest_prod.py
```

تم حفظ النتائج في `logs_prod/backtest_report.json` مع منحنيات الأسهم في ملف CSV.
