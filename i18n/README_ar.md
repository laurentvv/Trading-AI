## 🌟 حول المشروع

نظام خبير لدعم القرارات لتداول صناديق ETF لـNASDAQ والنفط (WTI)، يستفيد من ذكاء اصطناعي هجين مكون من 12 نموذجاً.

### 🚀 استراتيجية التذكرة المزدوجة (التحليل مقابل التداول)

يحلل النظام **مؤشراً عالمياً** (مثلاً `^NDX` لـNasdaq-100، `CL=F` لـWTI) لكنه **ينفذ على ETF مقوّم باليورو** (مثلاً `SXRV.DE`، `CRUDP.PA`). يضمن هذا الفصل تحليلاً على بيانات عالية الدقة وتنفيذاً فعلياً على أصول متاحة عبر Trading 212.

### 🧠 محرك الذكاء الاصطناعي الهجين

يجمع المحرك نماذج غير متجانسة في **إجماع مرجّح**:

1. **نماذج Scikit-Learn** (RandomForest، GradientBoosting، LogisticRegression) — مُتحقَّق منها عبر `TimeSeriesSplit` لمنع تسرّب البيانات. إشارة كمية عدوانية (25% من الوزن المعرفي).
2. **TimesFM 2.5** (Google Research) — نموذج أساسي للتنبؤ بالسلاسل الزمنية.
3. **TensorTrade / PPO** (stable-baselines3) — وكيل تعلم معزّز في بيئة Gymnasium مخصصة.
4. **Gemma 4 12B** (Ollama) — تحليل **نصي** (ماكرو/أخبار) و**بصري** (مخططات تقنية)؛ **دفاع JSON ثنائي الطبقة** يضمن JSON نظيفاً رغم تفعيل وضع التفكير `<|think|>`.
5. **تحليل المشاعر الهجين** (Alpha Vantage + AlphaEar + Hyperliquid).
6. **Vincent Ganne Model** — قفل جيوسياسي (WTI، Brent، غاز، يوريا، DXY) يُولّد إشارات BUY فقط للتحقق من قاع Nasdaq.
7. **OilBenchModel** — نموذج إدراكي متخصص في WTI (مؤشرات تقنية + أساسيات EIA + مشاعر).

### 🧘 فلسفة القرار: «الحذر المعرفي»

تحوز النماذج الإدراكية (Gemma 4، المشاعر، Vincent Ganne) على **75%** من وزن القرار مقابل **25%** للنموذج الكمي العدواني. تضمن هذه الترجيحة المتعمدة أن السياق النوعي يهدئ الإشارات الكمية. لا تُنفَّذ إشارة إلا إذا تجاوزت الثقة العالمية **40%**؛ بين 20% و40% تُخفَّض إلى HOLD.

### ✨ الميزات الرئيسية

- **معمارية LLM هجينة سحابية/محلية**: تكامل `free-llm-api-keys` للاستفادة من «نماذج Frontier» شديدة الذكاء (DeepSeek، Claude، Gemini) للتحليل النصي، مع احتياطي محلي 100% قوي إلى Ollama (الذي يبقى المحرك الحصري للمخططات البصرية).
- **نهج التذكرة المزدوجة**: حلّل المؤشر، تداول الـETF.
- **أسعار T212 الحية**: استرجاع أسعار اليورو لحظياً عبر واجهة Trading 212 (0.2 ثانية)، مع احتياطي yfinance وذاكرة parquet.
- **فارق Dated Brent**: مراقبة توتر السوق الفعلي عبر الفارق بين Brent Spot (Dated) وBrent الآجل.
- **مرونة الشبكة**: قاطع دارة yfinance بمتعقّبات منفصلة (info مقابل download)، مهلة 10 ثوانٍ على جميع نداءات الشبكة.
- **إبطال ذاكرة تلقائي**: تكتشف ذاكرة Parquet قِدَمها (> 2 يوم) وتفرض تحديثاً. استخدم `refresh_cache.py` للتفريغ اليدوي.
- **توازي نداءات LLM**: نداءات النماذج المستقلة (`text_llm`، `visual_llm`، `search_query`، `timesfm`، `tensortrade`، `grebenkov`) تعمل في `ThreadPoolExecutor` لتداخل استدلال Ollama مع I/O. المسار الحرج عادةً 4–6 دقائق على CPU مقابل 10+ دقيقة تسلسلياً.
- **ذاكرة استعلامات البحث لـ24 ساعة**: يُخزَّن استعلام البحث المُولَّد بواسطة LLM في `data_cache/search_queries/<ticker>_<date>_<price-sig>.json`. المفتاح حسب التاريخ + توقيع فعل السعر (تقسيم لوغاريتمي للإغلاق + bucket RSI)، لذا تغيير النظام يُبطّله. استعلامات الاحتياطي **لا** تُخزَّن أبداً (لا يمكن لفشل Ollama عابر أن يسمّم الذاكرة لمدة 24 ساعة).
- **مهلة دورة صارمة**: كل دورة تيكير مُغلَّفة في ميزانية 40 دقيقة (`CYCLE_TIMEOUT_SECONDS` في `main.py`). عند المهلة، يُغلق خيط العامل بـ `shutdown(wait=F)` ليبدأ التيكير التالي فوراً؛ يُطبَّق HOLD على التيكير المنتهي. لكل future مهلاته الخاصة (بحث 240ث، بصري 300ث، نص 240ث، نماذج CPU 180ث لكل منها، أخبار 90ث، زحف ويب 30ث).
- **أمان الخيط اليتيم**: عند مهلة الدورة، يُضبط `threading.Event` لكل تيكير حتى ينسحب العامل اليتيم قبل أي نداء `execute_t212_trade` — مانعاً صفقات بأموال حقيقية بعد أن رأى المستخدم لوحة «تم تطبيق HOLD». `threading.Lock` لكل تيكير يُسلسل أيضاً وضع أوامر T212، مزيلاً خطر الصفقة المزدوجة تحت تداخل المجدول أو نداءات `--ticker` المكررة.
- **حارس فشل LLM**: عندما يستنفد `_query_ollama` جميع إعادة المحاولات، يحمل قاموس الاحتياطي عَلَم `"failed": True` ليميّز منطق الإجماع下游 «النموذج اختار HOLD» عن «النموذج تحطم» (حالياً يُمرَّر لكن لا يُرشَّح — متابعة معروفة).
- **إدراك متقدم**: استخدام **Gemma 4 12B** مع **دفاع JSON ثنائي الطبقة**:
  1. **فرض المخطط على الخادم** (`format: SCHEMA_*` مع `additionalProperties: false`) — الطبقة الحاملة؛ تُمرَّر عبر معامل `format` لـOllama في كل موقع نداء. المخططات مُعرَّفة في `src/llm_client.py` (`SCHEMA_TRADING_DECISION`، `SCHEMA_SEARCH_QUERY`، `SCHEMA_OIL_ALLOCATION`).
  2. **لاحقة system prompt دفاعية** (`"...never add a 'thought' key."`) — خط ثاني زائد-لكن-غير-ضار، يُحتفظ به كـbelt-and-braces ضد أي انحدار مستقبلي لطبقة المخطط.

  رمز التفكير `<|think|>` **مُفعَّل** في جميع أوامر النظام الإنتاجية الأربعة (أُعيد تفعيله 2026-06-06 في `main` بعد التحقق على فرع `think-mode`). طبقة المخطط هي ما يُبطل فعلياً عيب حطام JSON التاريخي `<|channel>thought` (السبب الجذري مايو 2026): يؤكد `tests/check_llm_json.py` أن حالات schema-strict (`v3_schema`، `v6_schema`، `v7_schema_strict`) تُنتج JSON نظيفاً حتى مع تفعيل `<|think|>`، بينما متغيرات `format:json` الفضفاضة تفشل. للتحليل الكامل وإجراء التراجع راجع `docs/ADR-001-think-mode-dual-layer-defence.md`.
- **وكيل Morning Brief ذاتي**: تدفق ليلي قائم على `smolagents` (`morning_brief/morning_brief.py`) مُجدوَل تلقائياً في 01:00 عبر `schedule.py`. يزحف بشكل مستقل سجلات الـAPI اليومية، يحمّل بيانات مخزون EIA الأساسية، ويحكّم نقاش *Bull vs Bear*. يُحقن تقرير markdown الناتج (`morning_market_brief.md`) تلقائياً في system prompt الخاص بـLLM النصي خلال دورة التداول اليومية، مانحاً الذكاء الاصطناعي الرئيسي ذاكرة سياقية عميقة ووعياً أساسياً دون إبطاء التنفيذ على السوق الحي.
- **🏛️ Weekend Council (الذاكرة الاستراتيجية)**: مراجعة LLM أسبوعية متعددة الشخصيات (`src/council/weekend_council.py`) تعمل كل **سبت في 01:00** عبر `schedule.py`. ست شخصيات — كل واحدة على **عائلة نماذج Ollama مختلفة** (Gemma 4 12B / GLM-4.6V-Flash / Qwen 3.5 9B / LFM 2.5 / Mistral Nemo 12B) لتنوع استدلال حقيقي — تتداول عبر بروتوكول 4 جولات (بوابة إعادة صياغة المشكلة ← تحليل بـSTANCE صريح ← نقاش 1-ضد-1 ← تركيب القاضي) مع آليات مضادة للتفكير الجماعي (حصة معارضة، حكم غير-محلول-أولاً). يُصدر القاضي (Qwen3.5-9B-MTP) موقفاً لكل تيكير يصبح **التصويت المرجّح الحادي عشر** (9.5%) في إجماع الزمن الحقيقي، مع ثقة تتحلل خطياً على مدى 7 أيام. ميزانيات tokens سخية (`num_predict` حتى 12000، `num_ctx` حتى 65536) ونافذة مجدول 48 ساعة تستوعب نماذج التفكير على CPU. يحلل المجلس بيانات PROD الحقيقية: دقة النماذج (`model_performance.db`)، مقاييس المحفظة والتنبيهات الحرجة (`performance_monitor.db`)، ودفتر التداول المُنفَّذ. ثبّت الـ6 نماذج المطلوبة عبر `uv run python setup_council_models.py`. راجع `docs/ADR-003-weekend-council-11th-voice.md`.
- **الأخبار ومشاعر البلوكشين**: تكامل **AlphaEar** و**Hyperliquid** لالتقاط المشاعر الاجتماعية والمضاربية.
- **مجدول ذاتي**: سكربت `schedule.py` للتشغيل المستمر (8:30 – 18:00) على خادم.
- **إدارة مخاطر مركزية**: يُمركز `AdvancedRiskManager` منطق Anti-Loss (Stop-Loss) وTrailing Stop. لم تعد النماذج الفردية تدير هذه المخاطر، مما يضمن استراتيجية حماية رأس مال موحدة وصارمة عبر أنظمة السوق المختلفة.
- **عقود بيانات صارمة**: جميع نماذج الذكاء الاصطناعي مُوحدة بالكامل لإرجاع dataclass `ModelResult` مُنمَّطة بقوة (`signal`، `confidence`، `reasoning`)، مما يضمن تطابقاً 100% عبر محرك الإجماع.
- **صحة الكود مُدقَّقة**: يحافظ المشروع على معيار صحة كود **Grade B** عبر تدقيقات آلية (0 كود ميت، مؤشر قابلية صيانة عالٍ).
- **اختبار رجعي للإنتاج**: محرك اختبار رجعي مستقل (`backtest_prod.py`) يعيد تشغيل إشارات prod الحقيقية مقابل أسعار حقيقية برسوم T212 — بلا تبعيات خارجية.
- **التحكم في تفريغ التصحيح**: اضبط `TRADING_DEBUG_DUMP=0` لتعطيل تفريغ `data_cache/llm_debug_fail.txt` (المحدود بـ5 MB) لفشل LLM.

### 💻 الحزمة التقنية

- **اللغة**: `Python 3.12+`
- **الحسابات والبيانات**: `pandas`، `numpy`، `yfinance`، `pyarrow`، `pandas_datareader`، `hyperliquid-python-sdk`
- **تعلم الآلة**: `scikit-learn`، `shap`
- **الذكاء الاصطناعي وLLM**: `google-genai` (Gemini)، `requests`، `ollama`
- **كشط الويب والبحث**: `beautifulsoup4`، `duckduckgo_search`، `crawl4ai`
- **التصور**: `matplotlib` (خلفية Agg لسلامة الخيوط)، `seaborn`، `mplfinance`
- **أدوات**: `tqdm`، `rich`، `python-dotenv`، `schedule`

### ⚙️ الأداء والمعدات
صُمِّم النظام ليكون **عالي الأداء على معدات المستهلك** دون الحاجة إلى GPU مخصص.
- **CPU فقط**: استدلال LLM (Gemma 4 12B Q6_K عبر Ollama) وTimesFM يعملان بالكامل على CPU. الإنتاجية ~3–4 tokens/ث على CPU حديث بثمانية أنوية.
- **RAM الموصى به**: 16 GB كحد أدنى (32 GB موصى به لتشغيل Gemma 4 12B بسهولة مع TimesFM وTensorTrade).
- **تزامن Ollama**: اضبط `OLLAMA_NUM_PARALLEL=8` (موجود بالفعل في `.env` الموصى به) ليتمكن نداءات LLM المتعددة من مشاركة حمل النموذج. مع ميزانية السياق الافتراضية 4 GB، تحصل الخيوط المتوازية على ~512 token لكل منها — سيُسلسل Ollama إذا تجاوزت الـprompts الـctx لكل خيط، لكن `ThreadPoolExecutor` يبقى التداخل الزمني مفيداً للخطوات المرتبطة بـI/O (جلب الأخبار، زحف الويب، نماذج CPU).
- **زمن التنفيذ**: ~6 إلى 9 دقائق لكل تيكير على CPU (بارد)، ~3 إلى 5 دقائق لكل تيكير عند ضرب ذاكرة استعلامات البحث. افتراضياً يعمل تيكيران (CRUDP.PA + SXRV.DE)، فخطط لـ~15 دقيقة إجمالاً.
- **مهلة الدورة**: كل دورة تيكير محدودة بـ40 دقيقة (`CYCLE_TIMEOUT_SECONDS`). عند التجاوز، يُطبَّق HOLD ويبدأ التيكير التالي فوراً.
- **سرعة الـAPI**: تكامل Trading 212 فائق السرعة (<1 ثانية لاسترجاع السعر الحي).

### 🧠 معمارية الذكاء الاصطناعي و LLM (Gemini + احتياطي محلي)
يستفيد النظام من معمارية متعددة الطبقات قوية جداً لضمان أقصى فترة تشغيل وقرارات ذكية، مُدمَجة بعمق في `main.py` و Weekend Council.

- **تسلسل احتياطي من 4 طبقات**:
  1. **طبقة Gemini المدفوعة (`GEMINI_API_KEY_PAY`)**: الأولوية القصوى. تستخدم نماذج متقدمة مثل Gemini 2.5 Pro للاستدلال المعقد والرؤية التقنية للمخططات وقرارات التداول النهائية.
  2. **طبقة Gemini المجانية (`GEMINI_API_KEY`)**: تُستخدم للمهام الأخف عالية الحجم مثل تلخيص سياق الويب.
  3. **وكلاء LLM API المجانيون**: احتياطي عبر `free-llm-api-keys`.
  4. **Ollama المحلي**: احتياطي CPU دون اتصال 100% قوي إذا فشلت جميع الخدمات السحابية.
- **حماية التكلفة**: الطبقة المدفوعة محدوددة بميزانية تكلفة متدحرجة لـ30 يوماً (`GEMINI_PAY_MONTHLY_BUDGET_EUR`، افتراضي 8.6 €/شهر) — تُحتسب تكلفة كل نداء من استخدام token الفعلي × سعر النموذج وتُراكم؛ عند بلوغ الميزانية، تنتقل النداءات إلى الطبقة المجانية / Ollama. احتياطي يومي (`GEMINI_PAY_DAILY_CAP`، افتراضي 200) يحمي من الحلقات غير المنضبطة.
- **التكامل**: يستخدم محرك التنفيذ اليومي الرئيسي (`main.py`) Gemini لإجماع متعدد النماذج في الزمن الحقيقي، بينما يُدمج Weekend Council غير المتزامن (`council`) Gemini تحديداً لأدوار معينة (مثل القاضي والشكوك) إلى جانب نماذج Ollama المحلية المتنوعة.

### 🧠 FinAcumen (الذاكرة المالية)
دُمجت معمارية FinAcumen لمنح نماذج الذكاء الاصطناعي المحلية **ذاكرة خبرة** وأدوات حتمية. هذا يحل مشكلة فقدان الذاكرة في الـLLMs.
- يعمل FinAcumen **بشكل غير متزامن ليلاً** (عبر `schedule.py`) للاستفادة من القوة الكاملة للـCPU دون إعاقة دورات التداول.
- يُضاف تقريره النوعي العميق تلقائياً إلى **Morning Market Brief** لتوجيه LLM القرار طوال يوم التداول.

## 🚀 البداية السريعة

اتبع هذه الخطوات لإعداد بيئة التطوير المحلية لديك.

### ✅ المتطلبات المسبقة

- Python 3.12+ (عبر `uv`)
- [Ollama](https://ollama.com/) مُثبَّت ويعمل محلياً.
- نموذج LLM محمَّل: `ollama pull hf.co/unsloth/gemma-4-12b-it-GGUF:Q6_K`
- **نماذج Weekend Council** (اختيارية، لكنها مطلوبة لتنوع استدلال المجلس): يُشغّل المجلس كل شخصية على *عائلة* نماذج *مختلفة* (Gemma / GLM / Qwen / LFM). ثبّتها جميعاً دفعة واحدة عبر `uv run python setup_council_models.py`.

### ⚙️ التثبيت

1.  **استنساخ المستودع:**
    ```sh
    git clone https://github.com/laurentvv/Trading-AI.git
    cd Trading-AI
    ```
2.  **تثبيت `uv` (إن لم يكن مثبتاً):**
    راجع [astral.sh/uv](https://astral.sh/uv) لتعليمات التثبيت.

3.  **إنشاء وتفعيل البيئة الافتراضية (خطوة حاسمة):**
    يجب إنشاء وتفعيل `.venv` قبل تثبيت النماذج الأساسية.
    ```bash
    uv venv
    source .venv/bin/activate  # على Windows، استخدم `.\.venv\Scripts\activate.ps1`
    ```

4.  **تثبيت النماذج الأساسية:**
    شغّل سكربتات التثبيت لاستنساخ النماذج في `vendor/` وتطبيق الرقع:
    ```bash
    python setup_timesfm.py
    ```

5.  **تهيئة ومزامنة البيئة:**
    ```bash
    uv sync
    ```

6.  **تثبيت متصفحات لبحث الويب (Crawl4AI):**
    ```bash
    uv run python -m playwright install chromium
    ```

7.  **إعداد مفاتيح الـAPI لديك:**
    أنشئ ملف `.env` في جذر المشروع:
    ```
    ALPHA_VANTAGE_API_KEY="مفتاحك"
    EIA_API_KEY="مفتاحك"

    # اختياري لكن موصى به بشدة: تكامل Gemini AI
    GEMINI_API_KEY_PAY="مفتاح_الطبقة_المدفوعة"  # للاستدلال/الرؤية المعقدة (Gemini 2.5 Pro)
    GEMINI_API_KEY="مفتاح_الطبقة_المجانية"      # للمهام الأخف (تلخيص)
    GEMINI_PAY_MONTHLY_BUDGET_EUR=8.6        # ميزانية تكلفة متدحرجة لـ30 يوماً (€) — الحارس الحامل للفوترة
    GEMINI_PAY_DAILY_CAP=200                 # احتياطي: أقصى نداءات API مدفوعة يومياً
    ```

## 🛠️ الاستخدام

يُدرّب النظام نماذجه على أحدث البيانات عند كل تنفيذ قبل إصدار قرار.

### وضع المحاكاة (Paper Trading)

لاختبار النظام دون مخاطرة برأس مال افتراضي قدره 1000 €، استخدم علم `--simul`. سيُدير النظام سجلاً صارماً للمشتريات والمبيعات.

```sh
# شغّل تحليلاً محاكياً (الافتراضي: SXRV.DE - Nasdaq 100 EUR)
uv run main.py --simul

# شغّل على النفط (WTI)
uv run main.py --ticker CRUDP.PA --simul
```

### التنفيذ الفعلي (Trading 212)

أصبح النظام الآن **مُتكاملاً بالكامل** مع Trading 212:
- **التحقق من المحفظة**: قبل أي إجراء، يستشير الروبوت نقدك ومراكزك الحقيقية.
- **إدارة الـAPI**: يشمل آليات إعادة محاولة تلقائية ضد حدود الطلبات (Rate Limiting).

```sh
# شغّل التحليل مع تنفيذ فعلي (تجريبي أو حقيقي حسب .env)
uv run main.py --t212
```

## 🧪 الاختبار الرجعي في الإنتاج

يضم النظام **محرك اختبار رجعي إنتاجي مستقل** (`backtest_prod.py`) يعيد تشغيل إشارات prod الحقيقية من `logs_prod/trading_journal.csv` مقابل أسعار حقيقية من ملفات Parquet في `data_cache/`.

### الميزات
- **إشارات حقيقية**: يعيد تشغيل قرارات محرك الـ12 نموذجاً الهجين بدقة.
- **أسعار حقيقية**: يستخدم بيانات OHLCV الفعلية لـETFs (SXRV.DE، CRUDP.PA) — بلا وكلاء US.
- **رسوم T212**: يحاكي نموذج رسوم Trading 212 البالغ 0.1% لكل صفقة.
- **مقارنة الأساس**: يحسب أداء buy-and-hold تلقائياً كمعيار.
- **المقاييس**: نسبة Sharpe، أقصى Drawdown، Win Rate، Alpha، العائد الإجمالي لكل تيكير.

### الاستخدام

```bash
uv run python backtest_prod.py
```

تُحفظ النتائج في `logs_prod/backtest_report.json` مع منحنيات equity بصيغة CSV.
