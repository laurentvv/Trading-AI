## 🌟 О проекте

Этот проект представляет собой экспертную систему поддержки принятия решений для торговли ETF с использованием трехмодального гибридного подхода ИИ. Он разработан для обеспечения всестороннего и надежного анализа путем объединения нескольких перспектив ИИ.

### 🚀 Стратегия с двумя тикерами (Анализ против Торговли)
Система использует инновационный подход для максимизации точности моделей:
- **Высокоточный анализ**: Модели ИИ анализируют **глобальные эталонные индексы** (`^NDX` для Nasdaq, `CL=F` для сырой нефти WTI). Эти индексы предлагают более длинную историю и «более чистые» тренды, без шума, связанного с часами торговли или комиссиями ETF.
- **Исполнение ETF**: Реальные ордера размещаются на соответствующих тикерах в **Trading 212** (`SXRV.DE`, `CRUDP.PA`), используя **актуальные цены T212** (через API позиций) для определения размера позиции. Состояние портфеля синхронизируется напрямую из T212 (`sync_state_from_t212()`), а актуальные цены внедряются в конвейер анализа (`_inject_t212_live_price()` в `src/data.py`).

### 🧠 Гибридный ИИ-движок
Система объединяет одиннадцать различных сигналов:
1.  **Классическая количественная модель**: Ансамбль RandomForest/GradientBoosting/LogisticRegression, обученный на технических и макроэкономических индикаторах.
2.  **TimesFM 2.5 (Google Research)**: Передовая базовая модель для прогнозирования временных рядов.
3.  **TensorTrade / PPO (Обучение с подкреплением)**: Агент RL (stable-baselines3), обучающий политику PPO в пользовательской торговой среде Gymnasium с сохранением данных между циклами.
4.  **Модель Oil-Bench (Gemma 4 12B (Unsloth))**: Модель, специализирующаяся на энергетике, объединяющая фундаментальные данные **EIA** (Запасы, Импорт, Использование нефтеперерабатывающих заводов) и настроение рынка для торговли WTI.
5.  **Текстовая LLM (Gemma 4 12B (Unsloth))**: Контекстный анализ необработанных данных, новости в реальном времени через навык **AlphaEar**, и интеграция динамического **макроэкономического веб-исследования**. Она в явном виде потребляет ночной отчет **Morning Brief** для получения глубокой фундаментальной осведомленности перед принятием решений.
6.  **Визуальный LLM (Gemma 4 12B (Unsloth))**: Прямой анализ технических графиков (`enhanced_trading_chart.png`).
7.  **Анализ тональности (Sentiment Analysis)**: Гибридный анализ, сочетающий Alpha Vantage и «горячие» тренды от **AlphaEar** (Weibo, WallstreetCN).
8.  **Децентрализованные данные (Hyperliquid)**: Анализ спекулятивных настроений по нефти (WTI) через *Funding Rate* и *Open Interest*.
9.  **Модель Vincent Ganne**: Геополитический и кросс-активный анализ (WTI, Brent, Gas, DXY, MA200) для выявления макроэкономических минимумов.
10. **Модель Grebenkov**: Математическая модель следования за трендом, откалиброванная для кросс-активного анализа с использованием агностического паритета риска (Agnostic Risk Parity).
11. **Движок гибридного слияния**: Мета-модель, управляющая динамическим взвешиванием и когнитивным консенсусом между всеми подмоделями.

Цель состоит в том, чтобы выдать окончательное решение (`BUY`, `SELL`, `HOLD`) с абсолютным приоритетом **«Точность прежде всего» (Accuracy First)**.

### 🧘 Философия принятия решений: «Когнитивная осмотрительность»
В отличие от классических торговых алгоритмов, которые паникуют, как только возрастает волатильность, эта система применяет подход информированного инвестора:
- **Требуется сильный консенсус**: Количественная модель (Classic) может бить тревогу (`SELL`), но если когнитивные модели (Текстовый LLM, Визуальный, TimesFM) остаются нейтральными, система предпочтет `HOLD`.
- **Фильтр уверенности**: Решение о движении (покупка или продажа) подтверждается только в том случае, если общая уверенность превышает порог безопасности (обычно 40%). Ниже этого система рассматривает сигнал как «шум» и остается в режиме ожидания.
- **Защита капитала**: В режиме `VERY_HIGH` (очень высокий риск), `HOLD` служит в качестве щита. Это предотвращает вход на нестабильный рынок и позволяет избежать преждевременного выхода при простой технической коррекции, если фундаментальные факторы (News/Vision/Hyperliquid) не подтверждают неминуемый обвал.
- **Токен рассуждения `<|think|>`**: **Активен** во всех четырех системных промптах продакшена (включен обратно 2026-06-06 в ветке `main` после валидации в ветке `think-mode`). Слой схемы — это то, что реально нейтрализует исторический дефект JSON-мусора `<|channel>thought` (корневая причина мая 2026 г.): `tests/check_llm_json.py` подтверждает, что случаи со строгой схемой (`v3_schema`, `v6_schema`, `v7_schema_strict`) генерируют чистый JSON даже при включенном `<|think|>`, тогда как свободные варианты `format:json` завершаются ошибкой. Полный анализ и процедуру отката см. в `docs/ADR-001-think-mode-dual-layer-defence.md`.
- **Автономный агент Morning Brief**: Ночной процесс на базе `smolagents` (`morning_brief/morning_brief.py`), который автоматически запускается в 01:00 ночи через `schedule.py`. Он независимо анализирует ежедневные логи API, загружает фундаментальные данные по запасам EIA и проводит дебаты *Bull vs Bear*. Сгенерированный markdown-отчет (`morning_market_brief.md`) автоматически внедряется в системный промпт текстовой LLM во время дневного торгового цикла, давая главному ИИ глубокую контекстную память и фундаментальное понимание без замедления торговли в реальном времени.
- **Новостной и Blockchain Сентимент**: Интеграция **AlphaEar** и **Hyperliquid** для захвата социального и спекулятивного сентимента.

### ✨ Основные функции

- **Подход с двумя тикерами**: Анализ индекса, торговля ETF.
- **Актуальные цены T212**: Получение цен в EUR в реальном времени через API Trading 212 (0.2с), с резервным вариантом от yfinance и кешем parquet.
- **Спред Dated Brent**: Мониторинг напряжения физического рынка через спред между Brent Spot (Dated) и Brent Futures.
- **Сетевая отказоустойчивость**: Автоматический выключатель yfinance (circuit breaker) с отдельными трекерами (info vs. download), тайм-аут 10 с на все сетевые запросы.
- **Авто-аннулирование кеша**: Кеш parquet автоматически обнаруживает устаревание (> 2 дней) и принудительно обновляется. Используйте `refresh_cache.py` для ручной очистки кеша.
- **Распараллеливание вызовов LLM**: Независимые вызовы моделей (`text_llm`, `visual_llm`, `search_query`, `timesfm`, `tensortrade`, `grebenkov`) выполняются в `ThreadPoolExecutor` для перекрытия вывода Ollama с операциями ввода-вывода (I/O). Критический путь обычно занимает 4–6 мин на CPU против 10+ мин последовательно.
- **Кеш поисковых запросов на 24 часа**: Поисковый запрос для интернета, сгенерированный LLM, кэшируется в `data_cache/search_queries/<ticker>_<date>_<price-sig>.json`. Ключом является дата + сигнатура изменения цены (логарифмическое распределение цены закрытия + распределение RSI), поэтому смена режима аннулирует кеш. Резервные запросы **никогда** не кэшируются (один временный сбой Ollama не может "отравить" кеш на 24 часа).
- **Жесткий тайм-аут цикла**: Каждый цикл тикера ограничен бюджетом в 15 минут (`CYCLE_TIMEOUT_SECONDS` в `main.py`). По истечении тайм-аута рабочий поток завершается с помощью `shutdown(wait=False)`, поэтому следующий тикер запускается немедленно; к тикеру с истекшим временем применяется HOLD. Индивидуальные фьючерсы имеют свои собственные тайм-ауты задач (поиск 240с, визуальный 300с, текстовый 240с, модели CPU по 180с каждая, новости 90с, сканирование сети 30с).
- **Безопасность потерянных потоков (Orphan-Thread Safety)**: При тайм-ауте цикла устанавливается событие `threading.Event` для конкретного тикера, так что потерянный рабочий поток прекращает работу до вызова `execute_t212_trade` — предотвращая сделки с реальными деньгами после того, как пользователю была показана панель "HOLD appliqué". `threading.Lock` для каждого тикера дополнительно сериализует размещение ордеров T212, устраняя риск двойной сделки при перекрытии планировщика или дублировании вызовов `--ticker`.
- **Страж сбоев LLM**: Когда `_query_ollama` исчерпывает все попытки повторного запуска, резервный словарь содержит флаг `"failed": True`, чтобы логика консенсуса ниже по потоку могла отличить «модель выбрала HOLD» от «модель завершилась сбоем» (в настоящее время передается, но не фильтруется — известная задача на будущее).
- **Продвинутое познание**: Использование **Gemma 4 12B** с **двухслойной защитой JSON**:
  1. **Обеспечение схемы на стороне сервера** (`format: SCHEMA_*` с `additionalProperties: false`) — несущий слой; передается через параметр `format` в Ollama в каждом месте вызова. Схемы определены в `src/llm_client.py` (`SCHEMA_TRADING_DECISION`, `SCHEMA_SEARCH_QUERY`, `SCHEMA_OIL_ALLOCATION`).
  2. **Защитный суффикс системного промпта** (`"...never add a 'thought' key."`) — избыточная, но безвредная вторая линия защиты от любой будущей регрессии слоя схемы.

  Токен рассуждения `<|think|>` **активен** во всех четырех системных промптах в рабочей среде (снова включен 2026-06-06 в `main` после проверки в ветке `think-mode`). Слой схемы — это то, что фактически нейтрализует исторический дефект JSON-мусора `<|channel>thought` (коренная причина от мая 2026 года): `tests/check_llm_json.py` подтверждает, что случаи со строгой схемой (`v3_schema`, `v6_schema`, `v7_schema_strict`) выдают чистый JSON даже со включенным `<|think|>`, в то время как свободные варианты `format:json` терпят неудачу. См. `docs/ADR-001-think-mode-dual-layer-defence.md` для полного анализа и процедуры отката.
- **Настроения в новостях и блокчейне**: Интеграция **AlphaEar** и **Hyperliquid** для захвата социальных и спекулятивных настроений.
- **Автоматизированный планировщик**: Скрипт `schedule.py` для непрерывного выполнения (с 8:30 до 18:00) на сервере.
- **Централизованное управление рисками**: `AdvancedRiskManager` централизует логику предотвращения потерь (Stop-Loss) и скользящего стопа (Trailing Stop). Индивидуальные модели больше не управляют этими рисками, обеспечивая единую и строгую стратегию защиты капитала в различных рыночных режимах.
- **Строгие контракты данных**: Все модели ИИ полностью стандартизированы для возврата строго типизированного дата-класса `ModelResult` (`signal`, `confidence`, `reasoning`), обеспечивая 100% единообразие во всем механизме консенсуса.
- **Аудит работоспособности кода**: Проект поддерживает стандарт работоспособности кода **Grade B** с помощью автоматизированного аудита (0 мертвого кода, высокий индекс поддерживаемости).
- **Бэктестирование для продакшена**: Автономный движок бэктестирования (`backtest_prod.py`), воспроизводящий реальные производственные сигналы с реальными ценами и комиссиями T212 — без внешних зависимостей.
- **Контроль дампа отладки**: Установите `TRADING_DEBUG_DUMP=0` для отключения ограниченного (5 МБ) дампа сбоев LLM в `data_cache/llm_debug_fail.txt`.

### ⚙️ Производительность и оборудование
Система разработана так, чтобы быть **производительной на потребительском оборудовании**, не требуя выделенного графического процессора (GPU).
- **Только CPU**: Вывод LLM (Gemma 4 12B Q6_K через Ollama) и TimesFM полностью выполняются на CPU. Пропускная способность составляет ~3–4 токена/с на современном 8-ядерном CPU.
- **Рекомендуемая оперативная память**: минимум 16 ГБ (рекомендуется 32 ГБ для комфортной работы Gemma 4 12B наряду с TimesFM и TensorTrade).
- **Параллелизм Ollama**: Установите `OLLAMA_NUM_PARALLEL=8` (уже есть в рекомендуемом `.env`), чтобы несколько вызовов LLM могли разделять нагрузку на модель. При ограничении контекста в 4 ГБ по умолчанию параллельные слоты получают ~512 токенов каждый — Ollama будет сериализовать выполнение, если промпты превышают ctx на слот, но `ThreadPoolExecutor` сохраняет перекрытие (по реальному времени) полезным для шагов, ограниченных вводом-выводом (получение новостей, веб-сканирование, модели CPU).
- **Время выполнения**: ~6 - 9 минут на тикер на CPU (холодный старт), ~3 - 5 минут на тикер при попадании в кеш поисковых запросов. По умолчанию обрабатывается два тикера (CRUDP.PA + SXRV.DE), поэтому планируйте ~15 минут в сумме.
- **Тайм-аут цикла**: Каждый цикл тикера ограничен 15 минутами (`CYCLE_TIMEOUT_SECONDS`). В случае превышения применяется HOLD, и сразу же начинается обработка следующего тикера.
- **Скорость API**: Сверхбыстрая интеграция с Trading 212 (<1с для восстановления актуальной цены).

```

├── morning_brief/                   # Ночной автономный агент для глубокого фундаментального анализа

├── src/                             # Основные модули
│   ├── adaptive_weight_manager.py   # Динамическое взвешивание моделей на основе производительности
│   ├── advanced_risk_manager.py     # Управление рисками с учетом тренда и размеров позиций
│   ├── chart_generator.py           # Генерация технических графиков для визуального LLM
│   ├── classic_model.py             # Ансамбль количественных моделей Scikit-learn
│   ├── data.py                      # Получение, кэширование и предобработка данных
│   ├── database.py                  # Управление базой данных SQLite для метрик
│   ├── eia_client.py                # Клиент API Управления энергетической информации (EIA)
│   ├── enhanced_decision_engine.py  # Движок гибридного слияния, управляющий всеми моделями
│   ├── features.py                  # Разработка технических и макроэкономических признаков (feature engineering)
│   ├── grebenkov_model.py           # Математическая модель следования за трендом (Agnostic Risk Parity)
│   ├── llm_client.py                # Интеграция Ollama для локального вывода LLM
│   ├── news_fetcher.py              # Сканирование и разбор финансовых новостей
│   ├── oil_bench_model.py           # Специализированная торговая модель для WTI (энергетика)
│   ├── performance_monitor.py       # Отслеживание точности моделей и истории
│   ├── sentiment_analysis.py        # Интеграция настроений от Alpha Vantage и AlphaEar
│   ├── t212_executor.py             # Реальное исполнение и работа с портфелем через API Trading 212
│   ├── tensortrade_model.py         # Сигнал обучения с подкреплением (PPO)
│   ├── timesfm_model.py             # Интеграция прогнозирования временных рядов TimesFM 2.5
│   └── web_researcher.py            # Макроэкономическое веб-сканирование с Crawl4AI
```

## 🚀 Быстрый старт

Следуйте этим шагам для настройки вашей локальной среды разработки.

### ✅ Предварительные требования

- Python 3.12+ (через `uv`)
- [Ollama](https://ollama.com/), установленная и запущенная локально.
- Загруженная модель LLM: `ollama pull hf.co/unsloth/gemma-4-12b-it-GGUF:Q6_K`

### ⚙️ Установка

1.  **Клонируйте репозиторий:**
    ```sh
    git clone https://github.com/laurentvv/Trading-AI.git
    cd Trading-AI
    ```
2.  **Установите `uv` (если еще не установлено):**
    Смотрите [astral.sh/uv](https://astral.sh/uv) для инструкций по установке.

3.  **Создайте и активируйте виртуальное окружение (КРИТИЧЕСКИ ВАЖНЫЙ ШАГ):**
    Вы должны создать и активировать `.venv` перед установкой базовых моделей.
    ```bash
    uv venv
    source .venv/bin/activate  # В Windows используйте `.\.venv\Scripts\activate.ps1`
    ```

4.  **Установите базовые модели (Foundation Models):**
    Запустите скрипты установки, чтобы клонировать модели в `vendor/` и применить патчи:
    ```bash
    python setup_timesfm.py
    ```

5.  **Инициализируйте и синхронизируйте окружение:**
    ```bash
    uv sync
    ```

6.  **Установите браузеры для веб-исследований (Crawl4AI):**
    ```bash
    uv run python -m playwright install chromium
    ```

7.  **Настройте ваши ключи API:**
    Создайте файл `.env` в корне проекта:
    ```
    ALPHA_VANTAGE_API_KEY="YOUR_KEY"
    EIA_API_KEY="YOUR_KEY"
    ```

## 🛠️ Использование

Система обучает свои модели на самых свежих данных при каждом запуске перед тем, как вынести решение.

### Режим симуляции (Бумажная торговля)

Чтобы протестировать систему без риска с фиктивным капиталом в €1000, используйте флаг `--simul`. Система будет вести строгую историю покупок и продаж.

```sh
# Запустить симулированный анализ (По умолчанию: SXRV.DE - Nasdaq 100 EUR)
uv run main.py --simul

# Запустить для Нефти (WTI)
uv run main.py --ticker CRUDP.PA --simul
```

### Реальное исполнение (Trading 212)

В настоящее время система **полностью интегрирована** с Trading 212:
- **Проверка портфеля**: Перед любым действием робот сверяется с вашим реальным остатком средств и позициями.
- **Управление API**: Включает механизмы автоматических повторных запросов при превышении лимитов (Rate Limiting).

```sh
# Запустить анализ с реальным исполнением (Demo или Real в зависимости от .env)
uv run main.py --t212
```

## 🧪 Бэктестирование для продакшена

Система включает **автономный движок бэктестирования для продакшена** (`backtest_prod.py`), который воспроизводит фактические производственные сигналы из `logs_prod/trading_journal.csv` с реальными ценами из файлов Parquet в `data_cache/`.

### Возможности
- **Реальные сигналы**: Воспроизводит точные решения гибридного движка из 12 моделей.
- **Реальные цены**: Использует фактические данные ETF OHLCV (SXRV.DE, CRUDP.PA) — без использования американских прокси.
- **Комиссии T212**: Симулирует модель комиссий Trading 212 (0.1% за сделку).
- **Базовое сравнение (Baseline)**: Автоматически вычисляет производительность стратегии buy-and-hold (купить и держать) в качестве эталона.
- **Метрики**: Коэффициент Шарпа (Sharpe Ratio), Максимальная просадка (Maximum Drawdown), Процент выигрышей (Win Rate), Альфа (Alpha), Общая доходность (Total Return) на тикер.

### Использование

```bash
uv run python backtest_prod.py
```

Результаты сохраняются в `logs_prod/backtest_report.json` с CSV кривых доходности.
