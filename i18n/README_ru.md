## 🌟 О проекте

Экспертная система поддержки принятия решений для торговли ETF на NASDAQ и нефть (WTI), использующая гибридный искусственный интеллект из 12 моделей.

### 🚀 Стратегия Dual-Ticker (Анализ vs. Торговля)

Система **анализирует глобальный индекс** (напр. `^NDX` для Nasdaq-100, `CL=F` для WTI), но **торгует ETF в EUR** (напр. `SXRV.DE`, `CRUDP.PA`). Такое разделение обеспечивает анализ на высокодостоверных данных и реальную торговлю доступными через Trading 212 активами.

### 🧠 Гибридный ИИ-движок

Движок объединяет разнородные модели в **взвешенный консенсус**:

1. **Модели Scikit-Learn** (RandomForest, GradientBoosting, LogisticRegression) — валидированы через `TimeSeriesSplit` для предотвращения утечки данных. Агрессивный количественный сигнал (25 % когнитивного веса).
2. **TimesFM 2.5** (Google Research) — базовая модель для прогнозирования временных рядов.
3. **TensorTrade / PPO** (stable-baselines3) — RL-агент в кастомной среде Gymnasium.
4. **Gemma 4 12B** (Ollama) — **текстовый** (макро/новости) и **визуальный** (технические графики) анализ; **двухслойная JSON-защита** гарантирует чистый JSON при активном режиме размышления `<|think|>`.
5. **Гибридный анализ настроений** (Alpha Vantage + AlphaEar + Hyperliquid).
6. **Vincent Ganne Model** — геополитический засов (WTI, Brent, газ, мочевина, DXY), генерирующий только сигналы BUY для подтверждения оснований Nasdaq.
7. **OilBenchModel** — специализированная когнитивная модель для WTI (технические индикаторы + фундаментал EIA + настроения).

### 🧘 Философия принятия решений: «Когнитивная Осторожность»

Когнитивные модели (Gemma 4, настроения, Vincent Ganne) держат **75 %** веса решения против **25 %** агрессивной количественной модели. Это намеренное превышение гарантирует, что качественный контекст смягчает количественные сигналы. Сигнал исполняется только если глобальная уверенность превышает **40 %**; от 20 % до 40 % он понижается до HOLD.

### ✨ Ключевые возможности

- **Гибридная LLM-архитектура Cloud/Local**: интеграция `free-llm-api-keys` для использования очень умных «Frontier Models» (DeepSeek, Claude, Gemini) для текстового анализа со 100 % надёжным fallback на локальный Ollama (который остаётся эксклюзивным движком для визуальных графиков).
- **Подход Dual-Ticker**: анализировать индекс, торговать ETF.
- **Живые цены T212**: получение цен EUR в реальном времени через API Trading 212 (0,2 с) с fallback на yfinance и кэшем parquet.
- **Спред Dated Brent**: отслеживание напряжения физического рынка через спред между Brent Spot (Dated) и фьючерсным Brent.
- **Сетевая устойчивость**: автомат yfinance с раздельными трекерами (info vs. download), таймаут 10 с на все сетевые вызовы.
- **Авто-инвалидация кэша**: кэш Parquet обнаруживает устаревание (> 2 дней) и принудительно обновляется. Используйте `refresh_cache.py` для ручной очистки.
- **Параллелизация LLM-вызовов**: независимые вызовы моделей (`text_llm`, `visual_llm`, `search_query`, `timesfm`, `tensortrade`, `grebenkov`) выполняются в `ThreadPoolExecutor`, чтобы перекрыть инференс Ollama с I/O. Критический путь обычно 4–6 мин на CPU против 10+ мин последовательно.
- **24-часовой кэш поисковых запросов**: сгенерированный LLM поисковый запрос кэшируется в `data_cache/search_queries/<ticker>_<date>_<price-sig>.json`. Ключ по дате + сигнатуре ценового действия (логарифмическое buckets закрытия + bucket RSI), поэтому смена режима инвалидирует его. Запросы fallback **никогда** не кэшируются (один транзиентный сбой Ollama не может отравить кэш на 24 часа).
- **Жёсткий таймаут цикла**: каждый цикл по тикеру обёрнут в бюджет 15 мин (`CYCLE_TIMEOUT_SECONDS` в `main.py`). При таймауте рабочий поток `shutdown(wait=F)`, чтобы следующий тикер стартовал немедленно; к истёкшему тикеру применяется HOLD. Индивидуальные futures имеют собственные таймауты (поиск 240 с, визуал 300 с, текст 240 с, модели CPU по 180 с, новости 90 с, веб-краул 30 с).
- **Безопасность orphan-thread**: при таймауте цикла устанавливается `threading.Event` на тикер, чтобы orphan-воркер прервался до любого вызова `execute_t212_trade` — предотвращая реальные сделки после того, как пользователь увидел панель «HOLD применён». `threading.Lock` на тикер дополнительно сериализует размещение ордеров T212, устраняя риск двойной сделки при наложении scheduler или дублированных вызовах `--ticker`.
- **Сигнал сбоя LLM**: когда `_query_ollama` исчерпывает все повторные попытки, словарь fallback несёт маркер `"failed": True`, чтобы нижестоящая логика консенсуса могла отличить «модель выбрала HOLD» от «модель упала» (сейчас распространяется, но не фильтруется — известный follow-up).
- **Продвинутая когниция**: использование **Gemma 4 12B** с **двухслойной JSON-защитой**:
  1. **Принудительная схема на сервере** (`format: SCHEMA_*` с `additionalProperties: false`) — несущий слой; передаётся через параметр `format` Ollama на каждой точке вызова. Схемы определены в `src/llm_client.py` (`SCHEMA_TRADING_DECISION`, `SCHEMA_SEARCH_QUERY`, `SCHEMA_OIL_ALLOCATION`).
  2. **Защитный суффикс system prompt** (`"...never add a 'thought' key."`) — вторая избыточная-но-безопасная линия, сохранена как belt-and-braces против любой будущей регрессии слоя схемы.

  Токен размышления `<|think|>` **активен** во всех четырёх production system prompts (реактивирован 2026-06-06 в `main` после валидации на ветке `think-mode`). Именно слой схемы реально нейтрализует исторический дефект с мусором JSON `<|channel>thought` (первопричина май 2026): `tests/check_llm_json.py` подтверждает, что schema-strict случаи (`v3_schema`, `v6_schema`, `v7_schema_strict`) дают чистый JSON даже с активным `<|think|>`, тогда как loose-варианты `format:json` падают. См. `docs/ADR-001-think-mode-dual-layer-defence.md` для полного анализа и процедуры отката.
- **Автономный агент Morning Brief**: ночной workflow на `smolagents` (`morning_brief/morning_brief.py`), запускаемый автоматически в 01:00 через `schedule.py`. Он независимо обходит ежедневные логи API, загружает фундаментальные данные по запасам EIA и арбитрирует дебаты *Bull vs Bear*. Итоговый markdown-отчёт (`morning_market_brief.md`) автоматически внедряется в system prompt текстового LLM во время дневного торгового цикла, наделяя основную ИИ глубокой контекстной памятью и фундаментальной осведомлённостью без замедления исполнения на живом рынке.
- **🏛️ Weekend Council (Стратегическая память)**: еженедельная многоперсональная LLM-ретроспектива (`src/council/weekend_council.py`), запускаемая каждую **субботу в 01:00** через `schedule.py`. Шесть персон — каждая на **отдельном семействе моделей Ollama** (Gemma 4 12B / GLM-4.6V-Flash / Qwen 3.5 9B / LFM 2.5 / Mistral Nemo 12B) для подлинного разнообразия рассуждений — обсуждают по 4-раундовому протоколу (Problem Restate Gate → Analysis с явной STANCE → дебат 1-на-1 → синтез Судьи) с анти-groupthink механизмами (квота несогласия, вердикт unresolved-first). Судья (Qwen3.5-9B-MTP) выдаёт позицию по тикеру, которая становится **11-м взвешенным голосом** (9,5 %) консенсуса в реальном времени с линейным затуханием уверенности за 7 дней. Щедрые бюджеты токенов (`num_predict` до 12000, `num_ctx` до 65536) и 48-часовое окно scheduler accommodate мышление-модели на CPU. Council анализирует реальные PROD-данные: точность моделей (`model_performance.db`), метрики портфеля и критические оповещения (`performance_monitor.db`), а также исполненный торговый журнал. Установите 6 требуемых моделей через `uv run python setup_council_models.py`. См. `docs/ADR-003-weekend-council-11th-voice.md`.
- **Новости и блокчейн-настроения**: интеграция **AlphaEar** и **Hyperliquid** для захвата социального и спекулятивного настроя.
- **Автоматизированный scheduler**: скрипт `schedule.py` для непрерывной работы (8:30 – 18:00) на сервере.
- **Централизованное управление рисками**: `AdvancedRiskManager` централизует логику Anti-Loss (Stop-Loss) и Trailing Stop. Отдельные модели больше не управляют этими рисками, обеспечивая единую и строгую стратегию защиты капитала на разных рыночных режимах.
- **Строгие контракты данных**: все ИИ-модели полностью стандартизированы для возврата строго типизированной dataclass `ModelResult` (`signal`, `confidence`, `reasoning`), обеспечивая 100 % единообразие в движке консенсуса.
- **Аудит здоровья кода**: проект поддерживает стандарт здоровья кода **Grade B** через автоматизированные аудиты (0 мёртвого кода, высокий индекс сопровождаемости).
- **Бэктестинг продакшена**: автономный движок бэктеста (`backtest_prod.py`), воспроизводящий реальные prod-сигналы против реальных цен с комиссиями T212 — без внешних зависимостей.
- **Контроль debug-дампа**: установите `TRADING_DEBUG_DUMP=0`, чтобы отключить ограниченный (5 МБ) дамп `data_cache/llm_debug_fail.txt` сбоев LLM.

### ⚙️ Производительность и оборудование
Система спроектирована **производительной на потребительском оборудовании** без выделенного GPU.
- **Только CPU**: LLM-инференс (Gemma 4 12B Q6_K через Ollama) и TimesFM работают полностью на CPU. Пропускная способность ~3–4 токенов/с на современном 8-ядерном CPU.
- **Рекомендуемая RAM**: минимум 16 ГБ (32 ГБ рекомендуется для комфортного запуска Gemma 4 12B наряду с TimesFM и TensorTrade).
- **Конкурентность Ollama**: установите `OLLAMA_NUM_PARALLEL=8` (уже в рекомендуемом `.env`), чтобы несколько LLM-вызовов разделяли нагрузку модели. С бюджетом контекста по умолчанию 4 ГБ параллельные слоты получают ~512 токенов каждый — Ollama сериализует, если промпты превышают ctx на слот, но `ThreadPoolExecutor` сохраняет выгодное wall-clock перекрытие для I/O-связанных этапов (загрузка новостей, веб-краул, модели CPU).
- **Время выполнения**: ~6–9 минут на тикер на CPU (холодный), ~3–5 минут на тикер при попадании в кэш поисковых запросов. По умолчанию запускаются два тикера (CRUDP.PA + SXRV.DE), поэтому планируйте ~15 мин суммарно.
- **Таймаут цикла**: каждый цикл по тикеру ограничен 15 мин (`CYCLE_TIMEOUT_SECONDS`). При превышении применяется HOLD и следующий тикер стартует немедленно.
- **Скорость API**: сверхбыстрая интеграция Trading 212 (<1 с для получения живой цены).

### 🧠 Архитектура ИИ и LLM (Gemini + локальный fallback)
Система использует очень надёжную многоуровневую архитектуру для обеспечения максимального аптайма и интеллектуального принятия решений, глубоко интегрированную в `main.py` и Weekend Council.

- **4-уровневый каскадный fallback**:
  1. **Платный уровень Gemini (`GEMINI_API_KEY_PAY`)**: Высший приоритет. Использует продвинутые модели вроде Gemini 2.5 Pro для сложных рассуждений, технического зрения графиков и финальных торговых решений.
  2. **Бесплатный уровень Gemini (`GEMINI_API_KEY`)**: Используется для более лёгких высокообъёмных задач, таких как суммаризация веб-контекста.
  3. **Бесплатные прокси LLM API**: Резерв через `free-llm-api-keys`.
  4. **Локальный Ollama**: 100 % надёжный офлайн fallback на CPU, если все облачные сервисы падают.
- **Защита от расходов**: платный уровень ограничен бюджетом стоимости за 30 скользящих дней (`GEMINI_PAY_MONTHLY_BUDGET_EUR`, по умолчанию 8,6 €/мес) — стоимость каждого вызова вычисляется из фактического использования токенов × цены модели и накапливается; при достижении бюджета вызовы переходят на бесплатный уровень / Ollama. Ежедневный backstop (`GEMINI_PAY_DAILY_CAP`, по умолчанию 200) защищает от неконтролируемых циклов.
- **Интеграция**: основной движок ежедневного исполнения (`main.py`) использует Gemini для консенсуса многомоделей в реальном времени, тогда как асинхронный Weekend Council (`council`) интегрирует Gemini специфически для определённых ролей (таких как Судья и Скептик) наряду с разнообразными локальными моделями Ollama.

### 🧠 FinAcumen (Финансовая память)
Архитектура FinAcumen интегрирована, чтобы наделить локальные ИИ-модели **памятью опыта** и детерминированными инструментами. Это решает проблему амнезии LLM.
- FinAcumen работает **асинхронно ночью** (через `schedule.py`), чтобы использовать полную мощь CPU, не блокируя торговые циклы.
- Его глубокий качественный отчёт автоматически добавляется в **Morning Market Brief** для направления LLM решений на протяжении торгового дня.

## 📂 Структура проекта

Проект организован модульно для лучшей сопровождаемости.

```
Trading-AI/
├── morning_brief/                   # Ночной автономный агент для глубокого фундаментального анализа
│   ├── morning_brief.py             # Оркестратор агентов и конфигурация smolagents
│   └── output/                      # Генерируемые ежедневные markdown-отчёты (morning_market_brief.md)
├── src/                             # Основные модули
│   ├── adaptive_weight_manager.py   # Динамическое взвешивание моделей по производительности
│   ├── advanced_risk_manager.py     # Trend-Aware управление рисками и sizing
│   ├── bootstrap.py                 # Базовая логика инициализации
│   ├── chart_generator.py           # Генерирует технические графики для визуального LLM
│   ├── classic_model.py             # Ансамбль количественных моделей Scikit-learn
│   ├── config_weights.py            # Конфигурация базовых весов гибридного движка
│   ├── data.py                      # Получение, кэширование и предобработка данных
│   ├── database.py                  # Управление базой SQLite для метрик
│   ├── eia_client.py                # Клиент API Energy Information Administration
│   ├── enhanced_decision_engine.py  # Гибридный движок фузии, оркеструющий все модели
│   ├── enhanced_trading_example.py  # Примеры скриптов использования моделей
│   ├── features.py                  # Инженерия технических и макроэкономических признаков
│   ├── grebenkov_model.py           # Математическая Trend-Following модель (Agnostic Risk Parity)
```

## 🚀 Быстрый старт

Выполните эти шаги для настройки локального окружения разработки.

### ✅ Предварительные требования

- Python 3.12+ (через `uv`)
- [Ollama](https://ollama.com/) установлен и запущен локально.
- Загруженная LLM-модель: `ollama pull hf.co/unsloth/gemma-4-12b-it-GGUF:Q6_K`
- **Модели Weekend Council** (опционально, но требуются для разнообразия рассуждений council): council запускает каждую персону на *разном* семействе моделей (Gemma / GLM / Qwen / LFM). Установите их все сразу через `uv run python setup_council_models.py`.

### ⚙️ Установка

1.  **Клонировать репозиторий:**
    ```sh
    git clone https://github.com/laurentvv/Trading-AI.git
    cd Trading-AI
    ```
2.  **Установить `uv` (если ещё не сделано):**
    См. [astral.sh/uv](https://astral.sh/uv) для инструкций по установке.

3.  **Создать и активировать виртуальное окружение (КРИТИЧЕСКИЙ шаг):**
    Вы должны создать и активировать `.venv` перед установкой базовых моделей.
    ```bash
    uv venv
    source .venv/bin/activate  # На Windows используйте `.\.venv\Scripts\activate.ps1`
    ```

4.  **Установить базовые модели:**
    Запустите скрипты установки, чтобы клонировать модели в `vendor/` и применить патчи:
    ```bash
    python setup_timesfm.py
    ```

5.  **Инициализировать и синхронизировать окружение:**
    ```bash
    uv sync
    ```

6.  **Установить браузеры для веб-исследования (Crawl4AI):**
    ```bash
    uv run python -m playwright install chromium
    ```

7.  **Настроить ключи API:**
    Создайте файл `.env` в корне проекта:
    ```
    ALPHA_VANTAGE_API_KEY="ВАШ_КЛЮЧ"
    EIA_API_KEY="ВАШ_КЛЮЧ"

    # Опционально, но настоятельно рекомендуется: интеграция Gemini AI
    GEMINI_API_KEY_PAY="ВАШ_ПЛАТНЫЙ_КЛЮЧ"  # Для сложных рассуждений/зрения (Gemini 2.5 Pro)
    GEMINI_API_KEY="ВАШ_БЕСПЛАТНЫЙ_КЛЮЧ"    # Для более лёгких задач (суммаризация)
    GEMINI_PAY_MONTHLY_BUDGET_EUR=8.6        # Бюджет стоимости за 30 скользящих дней (€) — основной биллинг-гард
    GEMINI_PAY_DAILY_CAP=200                 # Backstop: макс. платных API-вызовов в день
    ```

## 🛠️ Использование

Система обучает свои модели на самых свежих данных при каждом запуске перед выдачей решения.

### Режим симуляции (Paper Trading)

Чтобы протестировать систему без риска с фиктивным капиталом 1000 €, используйте флаг `--simul`. Система будет вести строгую историю покупок и продаж.

```sh
# Запустить симулированный анализ (По умолчанию: SXRV.DE - Nasdaq 100 EUR)
uv run main.py --simul

# Запуск по нефти (WTI)
uv run main.py --ticker CRUDP.PA --simul
```

### Реальная торговля (Trading 212)

Система теперь **полностью интегрирована** с Trading 212:
- **Проверка портфеля**: перед любым действием робот сверяется с вашим реальным кэшем и позициями.
- **Управление API**: включает автоматические механизмы повторных попыток против лимитов запросов (Rate Limiting).

```sh
# Запуск анализа с реальным исполнением (Демо или Реал согласно .env)
uv run main.py --t212
```

## 🧪 Бэктестинг в продакшене

Система включает **автономный движок бэктеста продакшена** (`backtest_prod.py`), который воспроизводит реальные prod-сигналы из `logs_prod/trading_journal.csv` против реальных цен из файлов Parquet `data_cache/`.

### Возможности
- **Реальные сигналы**: воспроизводит точные решения гибридного движка из 12 моделей.
- **Реальные цены**: использует реальные OHLCV-данные ETF (SXRV.DE, CRUDP.PA) — без US-прокси.
- **Комиссии T212**: симулирует комиссионную модель Trading 212 0,1 % за сделку.
- **Сравнение baseline**: автоматически вычисляет эффективность buy-and-hold как бенчмарк.
- **Метрики**: коэффициент Шарпа, максимальная просадка, Win Rate, Alpha, суммарная доходность по тикеру.

### Использование

```bash
uv run python backtest_prod.py
```

Результаты сохраняются в `logs_prod/backtest_report.json` с кривыми equity в CSV.
