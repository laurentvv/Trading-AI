## Impact du déploiement

### 1. DB `trading_history.db` — Migration automatique
- La migration `_migrate_model_signals_table()` s'exécutera au premier `init_db()`.
- **La table `model_signals` a 0 rows** → la migration est safe (rien à perdre).
- La migration DROP/RENAME/CREATE/DROP va remplacer `'kronos'` par `'tensortrade'` dans le CHECK.
- **Risque** : La DB dans `logs_prod/` est un *snapshot* local — la vraie DB PROD est sur le serveur de prod. Vérifier que le serveur n'a pas de rows non plus.

### 2. `data_cache/tensortrade/` — Nouveau répertoire
- Le code crée `data_cache/tensortrade/ppo_model.zip` et `metadata.json` automatiquement.
- **Pas besoin de créer le répertoire manuellement** — `_MODEL_DIR.mkdir(parents=True, exist_ok=True)` le fait.
- **Pas besoin de nettoyer** quoi que ce soit dans `data_cache/` — les fichiers parquet existants et les modèles classic ne sont pas affectés.
- Premier cycle : entraînement initial (2000 steps, ~15-20 sec par ticker au lieu de ~5 sec).
- Cycles suivants : load + fine-tune (500 steps, ~4-6 sec par ticker).

### 3. `data_cache/models/` — Aucun impact
- Les 48 fichiers `classic_model_*.pkl` ne sont pas touchés par ce changement.
- Le nouveau `tensortrade/` sous-répertoire est séparé.

## Procédure de déploiement

### Étape 0 — Commit des changements
```bash
git add src/tensortrade_model.py src/database.py tests/
git commit -m "feat(tensortrade): PPO persistence, enriched env (10 features), DB integration"
git push
```

### Étape 1 — Arrêter le scheduler PROD
- Arrêter le processus `schedule.py` sur le serveur de prod.
- Attendre que le cycle en cours se termine (si actif — dure ~6 min max).

### Étape 2 — Pull sur le serveur PROD
```bash
git pull
```

### Étape 3 — Vérifier la DB PROD (lecture seule)
```bash
python -c "
import sqlite3
conn = sqlite3.connect('trading_history.db')
print(conn.execute('SELECT COUNT(*) FROM model_signals').fetchone())
print(conn.execute('SELECT sql FROM sqlite_master WHERE name=\"model_signals\"').fetchone())
"
```
- Si 0 rows → migration safe, passer à l'étape 4.
- Si > 0 rows → les données seront préservées par la migration (INSERT INTO ... SELECT ... WHERE model_type != 'kronos').

### Étape 4 — (Optionnel) Clean `data_cache/models/` obsolètes
Les 48 `classic_model_*.pkl` dans `data_cache/models/` sont probablement des caches pour des tickers qui ne sont plus actifs. **Ce n'est pas nécessaire pour ce déploiement**, mais si on veut nettoyer :
```bash
# DANGER: supprime les caches classic — le modèle se recréera au prochain cycle
# rm -rf data_cache/models/classic_model_*.pkl
```
**Recommandation** : Ne pas toucher. Le `classic_model` utilise un hash MD5 pour invalider les caches obsolètes.

### Étape 5 — Relancer le scheduler
```bash
python schedule.py
```

### Étape 6 — Vérifier le premier cycle
Surveiller les logs pour confirmer :
- `"Aucun modèle PPO en cache, entraînement initial..."` — attendu au premier cycle
- `"Modèle PPO initial sauvegardé (2000 timesteps)"` — confirmation de persistance
- Au cycle suivant : `"Modèle PPO chargé depuis le cache (2000 timesteps cumulés)"`

## Points de surveillance post-déploiement

| Signe | Action |
|-------|--------|
| `"Observation space changed, retraining from scratch"` dans les logs | Normal au 1er cycle. Si répété à chaque cycle → bug dans metadata. |
| `"Erreur chargement modèle PPO, retraining from scratch"` | Modèle corrompu — vérifier l'espace disque. Un re-entraînement se produit automatiquement. |
| Durée d'un cycle > 10 min | Le 1er cycle sera plus long (~20 sec TT par ticker). Si chronique, réduire `_INITIAL_TIMESTEPS`. |
| `data_cache/tensortrade/ppo_model.zip` > 50 MB | Improbable avec MlpPolicy, mais surveiller. |

## Risques et mitigations

| Risque | Probabilité | Mitigation |
|--------|-------------|------------|
| Migration DB échoue | Faible (0 rows à migrer) | Fallback automatique dans `_migrate_model_signals_table` (recovery from `_old`) |
| Premier cycle trop long | Moyen (~20 sec TT vs ~5 sec avant) | Acceptable dans un cycle de 6 min total |
| Modèle PPO corrompu sur disque | Faible | try/except avec fallback retrain from scratch |
| Espace disque insuffisant | Faible (~5 MB par modèle) | Un seul fichier `.zip`, écrasé à chaque cycle |
| Conflit d'accès concurrent au modèle | Nul | Un seul processus à la fois (scheduler séquentiel) |

## Rollback

Si problème en prod :
```bash
git revert HEAD
git push
python schedule.py  # redémarre avec l'ancien code
rm -rf data_cache/tensortrade  # optionnel, nettoie le nouveau répertoire
```
L'ancien code recréera le modèle from scratch à chaque cycle (comportement actuel).
