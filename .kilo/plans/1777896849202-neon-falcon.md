## 1. Analyse du Projet Trading-AI (Etat Actuel) <a name="1-analyse-du-projet"></a>

### 1.1 Profil du Systeme
- **Type** : Systeme d'aide a la decision hybride multi-modal (9 signaux IA)
- **Actifs** : ETF Nasdaq (SXRV.DE) et Petrole/WTI (CRUDP.PA)

## 2. Analyse de QuantConnect Lean <a name="2-analyse-de-lean"></a>

### 2.1 Architecture Interne de Lean

Lean est un moteur de trading algorithmique event-driven, principalement ecrit en **C#** (94.1%), avec support Python (5.7%) via **Python.NET**.

**Modules cles (prioritaires pour Trading-AI) :**

```
Lean/
├── Engine/                          # Moteur principal
│   ├── DataFeeds/                   # Gestion des flux de donnees
│   │   ├── FileSystemDataFeed       # Donnees locales (ZIP/CSV)
│   │   └── DownloaderDataProvider   # Telechargement automatique
│   ├── HistoricalData/              # Donnees historiques
│   ├── Results/                     # Resultats de backtest
│   ├── TransactionHandlers/         # Gestion des transactions
│   └── Setup/                       # Initialisation
├── Algorithm.Framework/             # Framework Alpha-Portfolio-Risk-Execution
│   ├── Alphas/                      # Modeles de signaux (Alpha Models)
│   ├── Portfolio/                   # Construction de portefeuille
│   ├── Risk/                        # Gestion des risques
│   └── Execution/                   # Modeles d'execution
├── Indicators/                      # 200+ indicateurs techniques
│   ├── RSI, MACD, Bollinger, ATR   # Indicateurs classiques
│   ├── CandlestickPatterns/         # 60+ patterns chandeliers
│   └── Greeks, IV, Hurst           # Indicateurs avances
├── Brokerages/                      # Couche d'abstraction courtier
│   ├── Backtesting/                 # Simulateur de backtest
│   └── Paper/                       # Paper trading
├── Common/                          # Types communs, Securities
├── Data/                            # Format de donnees Lean
├── Optimizer/                       # Optimisation de parametres
├── Report/                          # Rapports de performance
└── Algorithm.Python/                # Exemples d'algorithmes Python
```

### 2.2 Le Framework Algorithm (Alpha → Portfolio → Risk → Execution)

Le module `Algorithm.Framework` est le coeur de Lean et correspond exactement a l'architecture dont Trading-AI a besoin :

```
┌──────────────────┐    ┌──────────────────────┐    ┌──────────────────┐    ┌───────────────┐
│  Universe        │───>│  Alpha Models         │───>│  Portfolio       │───>│  Execution    │
│  Selection       │    │  (Signaux)            │    │  Construction    │    │  Model        │
└──────────────────┘    └──────────────────────┘    └──────────────────┘    └───────────────┘
                               │                            │                       │
                        ┌──────┴──────┐              ┌──────┴──────┐         ┌──────┴──────┐
                        │ Risk Mgmt   │              │ Position    │         │ Fill Model  │
                        │ (Overrides) │              │ Sizing      │         │ (Slippage)  │
                        └─────────────┘              └─────────────┘         └─────────────┘
```

**Mapping avec Trading-AI :**

| Module Lean | Equivalent Trading-AI | Amelioration |
|---|---|---|
| `Alpha Model` | `EnhancedDecisionEngine` (9 modeles) | Structure formelle Insight/Direction/Magnitude |
| `Portfolio Construction` | Kelly Criterion dans `AdvancedRiskManager` | Optimisation multi-positions, Black-Litterman |
| `Risk Management` | `AdvancedRiskManager` | Framework composable, MaximumDrawdownPercent |
| `Execution Model` | `t212_executor.py` | Fill simulation, slippage, latency modeling |
| `Universe Selection` | Hardcoded (SXRV.DE, CRUDP.PA) | Univers dynamique, filtrage fondamental |

## 3. Cas d'Usage Concrets pour Trading-AI <a name="3-cas-dusage"></a>

### 3.1 Cas d'Usage #1 : Backtesting Institutionnel (PRIORITE HAUTE)

**Probleme actuel** : Le backtest actuel de Trading-AI est simpliste :
- Decisions hebdomadaires uniquement
- Pas de slippage, pas de latence, pas de spread bid-ask
- Pas de fill simulation
- Resultats non comparables a des benchmarks institutionnels

**Ce que Lean apporte** :
- **Backtesting Engine** complet avec fill simulation realiste
- **Slippage models** (constant, volume-share, custom)
- **Fee models** personnalisables (0.1% par trade pour T212)
- **Split/dividend adjustments** automatiques
- **Warm-up periods** pour les indicateurs
- **Reporting** : Sharpe, Sortino, Max Drawdown, Alpha/Beta, tracking error

**Application** : Remplacer le backtest basique par un backtest Lean en mode "white-glove", en injectant les 9 signaux IA de Trading-AI comme `Alpha Models` composites.

### 3.2 Cas d'Usage #2 : Validation des Signaux IA (PRIORITE HAUTE)

**Probleme** : Pas de moyen systematique de valider chaque modele individuellement sur historique.

**Ce que Lean apporte** :
- Chaque modele IA peut etre encapsule comme un `Alpha Model` independant
- Le `CompositeAlphaModel` permet de tester chaque signal individuellement
- **Insight scoring** : chaque prediction est tracee avec direction, magnitude, confidence, et score final
- **Regression testing** automatise via le framework de tests de Lean

**Application** : Creer 9 `AlphaModel` Python (Classic, TimesFM, TensorTrade, LLM Text, LLM Visual, Sentiment, Hyperliquid, OilBench, VincentGanne) et les tester independamment avant fusion.

### 3.3 Cas d'Usage #3 : Donnees Multi-Source Fiables (PRIORITE MOYENNE)

**Probleme** : yfinance est instable, pas de donnees intraday fiables.

**Ce que Lean apporte** :
- **FileSystemDataFeed** : Lecture de donnees locales (format Lean)
- **DownloaderDataProvider** : Telechargement automatique depuis QuantConnect
- Support natif de **tick, second, minute, hour, daily** resolutions
- **Consolidators** : Agregation automatique de bars (tick → minute → hour → daily)
- **Alternative data** : EIA, FRED, custom data feeds

**Application** : Utiliser les DataFeeds de Lean comme source de donnees primaire, en gardant yfinance/yfinance comme fallback.

### 3.4 Cas d'Usage #4 : Risk Management Institutionnel (PRIORITE MOYENNE)

**Ce que Lean apporte** :
- `MaximumDrawdownPercentPerSecurity` (deja equivalent partiel dans Trading-AI)
- `MaximumSectorExposureRiskManagementModel`
- `TrailingStopRiskManagementModel`
- `CompositeRiskManagementModel` (composition de plusieurs risk managers)
- Framework composable et testable

### 3.5 Cas d'Usage #5 : Paper Trading Avance (PRIORITE BASSE)

Le paper trading de Lean (`Brokerages/Paper/`) est plus sophistique que la simulation SQLite actuelle :
- Fill simulation avec spread bid-ask
- Latence simulee
- Journal complet des ordres

### 3.6 Cas d'Usage NON Applicables

| Fonctionnalite Lean | Pourquoi Non Applicable |
|---|---|
| **Brokerage Trading 212** | Aucun support natif (Lean supporte IBKR, Alpaca, Bybit, Binance, etc.) |
| **Options/Futures trading** | Trading-AI ne trade que des ETFs spot |
| **Cloud deployment (QuantConnect)** | Trading-AI tourne en local sur Windows |
| **Universe selection dynamique** | Trading-AI trade 2 ETFs fixes |
| **Live trading via Lean** | L'execution T212 est deja fonctionnelle |

### 4.2 Modules Lean a Exploiter (Par Priorite)

#### Priorite 1 : Moteur de Backtesting (`Engine/`, `Brokerages/Backtesting/`)
- **Pourquoi** : Le backtest actuel est le plus gros gap fonctionnel
- **Modules** : `Engine/Engine.cs`, `Engine/Results/`, `Brokerages/Backtesting/`
- **Usage** : Runner des backtests historiques avec les signaux IA pre-calculules

#### Priorite 2 : Module Indicateurs (`Indicators/`)
- **Pourquoi** : Remplacer le calcul manuel dans `src/features.py`
- **Modules** : RSI, MACD, Bollinger, ATR, ADX, Stochastic, Ichimoku, etc.
- **Usage** : Standardiser les indicateurs techniques avec une bibliotheque eprouvee

#### Priorite 3 : Framework Alpha (`Algorithm.Framework/`)
- **Pourquoi** : Structurer formellement les 9 modeles IA
- **Modules** : `Alphas/`, `Portfolio/`, `Risk/`, `Execution/`
- **Usage** : Valider chaque modele independamment, optimiser les poids

#### Priorite 4 : Rapports (`Report/`)
- **Pourquoi** : Generer des rapports institutionnels
- **Usage** : Sharpe ratio, maximum drawdown, alpha/beta, tracking error

#### Priorite 5 : Optimiseur (`Optimizer/`)
- **Pourquoi** : Optimiser les poids des modeles et seuils de confiance
- **Usage** : Grid search ou bayesian optimization sur les hyperparametres

### 4.3 Modules a NE PAS Utiliser

| Module | Raison |
|---|---|

| `Data/` (QuantConnect data) | Necessite un compte cloud |
| `Api/` | Cloud-only |
| `Queues/` | Architecture cloud |
| `Messaging/` | Not needed |

## 5. Avantages et Limites <a name="5-avantages-limites"></a>

### 5.1 Avantages pour Trading-AI

| Avantage | Impact | Detail |
|---|---|---|
| **Backtesting institutionnel** | CRITIQUE | Fill simulation, slippage, fees, split adjustments |
| **200+ indicateurs natifs** | ELEVE | Remplace `src/features.py` manuel |
| **Alpha Framework** | ELEVE | Structure formelle pour les 9 modeles |
| **Reporting avance** | ELEVE | Sharpe, Sortino, MaxDD, Alpha, Beta |
| **Open-source, Apache 2.0** | MOYEN | Pas de cout de licence |
| **Communaute active** | MOYEN | 18.8k stars, 4.7k forks, documentation riche |
| **Multi-resolution** | MOYEN | Tick, second, minute, hourly, daily |
| **Optimiseur integre** | MOYEN | Grid search, bayesian optimization |

### 5.2 Limites Techniques

| Limite | Severite | Mitigation |
|---|---|---|
| **Principalement C#** | HAUTE | Python support via Python.NET, mais bindings parfois incomplets. Necessite .NET 9 SDK |
| **Pas de support T212** | HAUTE | Garder `t212_executor.py` pour l'execution live. Lean uniquement pour backtesting |
| **Courbe d'apprentissage** | HAUTE | Architecture event-driven specifique. Modeles C# a comprendre pour debug |
| **Python.NET overhead** | MOYENNE | Performances reduites vs C# natif. Problemes de serialisation types Python/C# |
| **Format de donnees proprietaire** | MOYENNE | Lean utilise son propre format binaire. Convertisseurs disponibles |
| **Pas de ML/AI natif** | MOYENNE | Lean ne supporte pas nativement LLMs, TimesFM, RL. Integration custom necessaire |
| **Setup complexe** | MOYENNE | Necessite .NET SDK, Python 3.11 specifique, variable d'env PYTHONNET_PYDLL |
| **Windows compatibilite** | BASSE | Supporte mais Docker recommande |
| **Pas de donnees gratuites intraday** | MOYENNE | QuantConnect data necessite un compte. yfinance reste le fallback |

### 5.3 Courbe d'Apprentissage Estimee

| Phase | Duree Estimee | Description |
|---|---|---|
| **Phase 1 : Setup & Comprehension** | 1-2 semaines | Installation .NET + Lean, comprendre l'architecture event-driven, lancer un backtest simple en Python |
| **Phase 2 : Premier Alpha Model** | 1 semaine | Encapsuler le modele Classic (scikit-learn) comme Alpha Model Lean en Python |
| **Phase 3 : Integration Multi-Modeles** | 2-3 semaines | Porter les 9 modeles en Alpha Models composites |
| **Phase 4 : Backtesting Complets** | 1-2 semaines | Configurer fees/slippage, generer rapports institutionnels |
| **Phase 5 : Optimisation** | 2-4 semaines | Grid search sur poids/parametres, validation hors-echantillon |

**Total estime : 7-12 semaines** a raison de 50% du temps de developpement.

### 5.4 Verdict Final

**Recommandation : Integration selective et progressive.**

Lean apporte une valeur considerable sur le **backtesting** et la **validation des modeles**, mais l'architecture C# dominante et l'absence de support T212 rendent une migration complete contre-productive. L'approche "Lean Bridge" (Lean comme couche de validation, Trading-AI comme moteur de production) est la strategie optimale.

```python
"""
Bridge pour injecter les signaux pre-calculules de Trading-AI
dans le format de donnees Lean.
"""
import json
import pandas as pd
from pathlib import Path
from datetime import datetime


class LeanSignalBridge:
    """Convertit les journaux Trading-AI en format Lean-compatible."""

    def __init__(self, journal_path: str = "trading_journal.csv"):
        self.journal_path = Path(journal_path)

    def journal_to_lean_signals(self) -> pd.DataFrame:
        """Lit le CSV journal et produit un DataFrame de signaux."""
        if not self.journal_path.exists():
            raise FileNotFoundError(f"Pas de journal: {self.journal_path}")

        df = pd.read_csv(self.journal_path)
        df["Timestamp"] = pd.to_datetime(df["Timestamp"])

        signals = []
        for _, row in df.iterrows():
            signals.append({
                "date": row["Timestamp"],
                "ticker": row["Ticker"],
                "signal": row["FINAL_SIGNAL"],
                "confidence": float(row["Confidence"].replace("%", "")) / 100,
                "risk_level": row["Risk_Level"],
                "risk_adjusted": row["Risk_Adjusted"],
            })
        return pd.DataFrame(signals)

    def export_to_lean_format(self, output_dir: str = "lean_data"):
        """Exporte les signaux en fichiers CSV format Lean."""
        out = Path(output_dir)
        out.mkdir(exist_ok=True)
        signals = self.journal_to_lean_signals()
        for ticker in signals["ticker"].unique():
            ticker_signals = signals[signals["ticker"] == ticker]
            filename = f"{ticker.replace('.', '_')}_signals.csv"
            ticker_signals.to_csv(out / filename, index=False)
```

from AlgorithmImports import *
from typing import List, Dict


class ClassicAlphaModel(AlphaModel):
    """Alpha Model encapsulant le modele scikit-learn de Trading-AI."""

    def __init__(self, lookback: int = 252):
        self.lookback = lookback

    def update(self, algorithm: QCAlgorithm, data: Slice) -> List[Insight]:
        insights = []

        for symbol in algorithm.securities.keys():
            # Recuperer l'historique via Lean (remplace yfinance)
            history = algorithm.history(symbol, self.lookback, Resolution.DAILY)

            if history.empty or len(history) < 60:
                continue

            close = history["close"]

            # Indicateurs natifs Lean (remplacent src/features.py)
            returns = close.pct_change().dropna()
            volatility = returns.std() * (252 ** 0.5)

            # Signal simplifie basé sur momentum + volatilite
            momentum_20 = (close.iloc[-1] / close.iloc[-20]) - 1
            momentum_50 = (close.iloc[-1] / close.iloc[-50]) - 1

            # Determiner direction
            if momentum_20 > 0.02 and momentum_50 > 0:
                direction = InsightDirection.UP
                magnitude = min(0.05, abs(momentum_20))
            elif momentum_20 < -0.02 and momentum_50 < 0:
                direction = InsightDirection.DOWN
                magnitude = min(0.05, abs(momentum_20))
            else:
                direction = InsightDirection.FLAT
                magnitude = 0.0

            confidence = max(0.1, 1.0 - volatility * 5)

            insights.append(Insight.price(
                symbol,
                timedelta(days=7),
                direction,
                magnitude,
                confidence,
                "ClassicAlpha"
            ))

        return insights


class TradingAICompositeAlpha(AlphaModel):
    """
    Alpha composite qui simule la fusion des 9 modeles Trading-AI.
    En production, chaque sous-modele peut etre active independamment.
    """

    def __init__(self):
        self.classic_alpha = ClassicAlphaModel()
        self.weights = {
            "classic": 0.10,
            "timesfm": 0.20,
            "llm_text": 0.15,
            "llm_visual": 0.10,
            "sentiment": 0.10,
            "vincent_ganne": 0.15,
            "oil_bench": 0.10,
            "tensortrade": 0.10,
        }

    def update(self, algorithm: QCAlgorithm, data: Slice) -> List[Insight]:
        return self.classic_alpha.update(algorithm, data)

    """
    Algorithme Lean Framework complet utilisant les Alpha Models Trading-AI.
    """

    def initialize(self):
        self.set_start_date(2020, 1, 1)
        self.set_end_date(2025, 12, 31)

        self.set_account_currency("EUR")

        # Actifs (proxies US pour les ETFs EUR de Trading-AI)
        symbols = [
            Symbol.create("QQQ", SecurityType.EQUITY, Market.USA),
            Symbol.create("USO", SecurityType.EQUITY, Market.USA),
        ]

        # Framework components
        self.set_universe_selection(ManualUniverseSelectionModel(symbols))
        self.set_alpha(TradingAICompositeAlpha())
        self.set_portfolio_construction(
            EqualWeightingPortfolioConstructionModel(Resolution.DAILY)
        )
        self.set_risk_management(
            MaximumDrawdownPercentPerSecurity(0.15)
        )
        self.set_execution(ImmediateExecutionModel())

    def on_order_event(self, order_event):
        if order_event.status == OrderStatus.FILLED:
            self.log(f"ORDER: {order_event.symbol} {order_event.direction} "
                     f"@ {order_event.fill_price:.2f} x{order_event.fill_quantity}")

```python
"""
Module de validation automatique : a chaque modification du Decision Engine,
relance un backtest Lean pour valider les performances.
"""
import subprocess
import json
from pathlib import Path


class LeanValidator:
    """Valide les changements de Trading-AI via backtest Lean."""

    def __init__(self, lean_project_path: str = "TradingAI-Lean"):
        self.lean_path = Path(lean_project_path)

    def run_backtest(self, start_year: int = 2020, end_year: int = 2025) -> dict:
        """Lance un backtest Lean et retourne les metriques cles."""
        result = subprocess.run(
            ["lean", "backtest", "--output", "validation-results"],
            capture_output=True,
            text=True,
            cwd=str(self.lean_path),
            timeout=600,
        )

        report_file = self.lean_path / "validation-results" / "report.json"
        if report_file.exists():
            with open(report_file) as f:
                report = json.load(f)
            return {
                "sharpe": report.get("Sharpe Ratio", 0),
                "max_drawdown": report.get("Max Drawdown", 0),
                "total_return": report.get("Total Return", 0),
                "win_rate": report.get("Win Rate", 0),
                "total_trades": report.get("Total Trades", 0),
            }
        return {"error": result.stderr}

    def validate_change(self, min_sharpe: float = 0.5, max_drawdown: float = 0.25) -> bool:
        """Valide qu'un changement ne degrade pas les performances."""
        metrics = self.run_backtest()

        passed = True
        if metrics.get("sharpe", 0) < min_sharpe:
            print(f"FAIL: Sharpe {metrics['sharpe']:.2f} < {min_sharpe}")
            passed = False
        if abs(metrics.get("max_drawdown", 0)) > max_drawdown:
            print(f"FAIL: MaxDD {abs(metrics['max_drawdown']):.2%} > {max_drawdown:.2%}")
            passed = False

        if passed:
            print(f"PASS: Sharpe={metrics['sharpe']:.2f}, "
                  f"Return={metrics['total_return']:.2%}, "
                  f"MaxDD={abs(metrics['max_drawdown']):.2%}")

        return passed
```

| Alternative | Avantage | Inconvenient |
|---|---|---|

| **Remain custom** | Zero overhead | Pas de validation independante |

**Recommandation** : Si le setup .NET/Python.NET bloque, migrer vers **Backtrader** comme plan B pour le backtesting uniquement, tout en gardant Trading-AI comme moteur de production.
