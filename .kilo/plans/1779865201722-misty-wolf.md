## 2. Fallback News quand Alpha Vantage est rate-limited

### Problème
`src/news_fetcher.py` : Quand Alpha Vantage est rate-limited (quasi-systématiquement en PROD), le fallback AlphaEar retourne surtout des news chinoises, et le score sentiment = 0.0 si aucune news n'est collectée.

### Solution : Ajouter un fallback Finnhub (gratuit, pas de clé nécessaire pour les news generales)

**`src/news_fetcher.py`** :
- Ajouter une fonction `fetch_finnhub_news(ticker: str)` qui utilise l'API Finnhub `https://finnhub.io/api/v1/news?category=general` (pas de clé requise pour le endpoint general) ou `https://finnhub.io/api/v1/company-news?symbol=AAPL&from=...&to=...` (clé gratuite)
- En fait, Finnhub requiert une clé. Mieux : utiliser **Google News RSS** qui ne nécessite aucune clé :
  - URL : `https://news.google.com/rss/search?q={query}+oil+commodity&hl=en-US&gl=US&ceid=US:en`
  - Parser le XML avec `xml.etree.ElementTree` (stdlib)
- Ajouter la fonction `fetch_google_news_rss(query: str, max_items: int = 10) -> tuple[list[str], float]`
- Modifier le `__main__` pour appeler ce fallback quand Alpha Vantage retourne 0 headlines
- Logique : si `av_headlines` est vide → appeler `fetch_google_news_rss` avec les mêmes topics → merger

### Fichiers modifiés
- `src/news_fetcher.py` : Ajouter `fetch_google_news_rss()` + logique de fallback dans `__main__`

## 3. Correction des poids adaptatifs (model_performance.db jamais rempli)

### Problème root cause
Les outcomes (`actual_outcome`) ne sont mis à jour **que lors d'un SELL confirmé** dans `t212_executor.py`. En mode simulation ou quand le système ne fait que du HOLD/BUY, les predictions restent à `actual_outcome = NULL` indéfiniment → `min_observations` (10) jamais atteint → poids jamais adaptés.

### Solution : Résolution automatique des outcomes au cycle suivant

**`src/enhanced_trading_example.py`** :
- Après la collecte des données de marché, **avant** `calculate_adaptive_weights`, ajouter un appel à une nouvelle méthode `resolve_previous_predictions(ticker, current_price, hist_data)` qui :
  1. Récupère les dates où `actual_outcome IS NULL` dans `model_performance_history`
  2. Pour chaque date, calcule le retour réel sur 1 jour en comparant le prix à J et J+1 dans `hist_data`
  3. Met à jour `actual_outcome` et `return_1d` pour toutes les predictions de cette date

**`src/adaptive_weight_manager.py`** :
- Ajouter la méthode `resolve_previous_predictions(dates_prices: dict)` qui prend un dict `{date: (price_at_date, price_next_day)}` et fait le UPDATE en batch
- Cette méthode est appelée par `enhanced_trading_example.py` au début de chaque cycle avec les données de prix disponibles

### Fichiers modifiés
- `src/adaptive_weight_manager.py` : Ajouter `resolve_previous_predictions(dates_prices)`
- `src/enhanced_trading_example.py` : Appeler la résolution avant le calcul des poids adaptatifs
